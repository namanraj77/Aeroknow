ROS2 Core Concepts — The Mental Model

Before writing any code, you need a clear mental model of how ROS2 works. This section covers exactly as much theory as you need — no more. Every concept here will be directly used in a step below.

The Four Communication Patterns

Topic (Publish / Subscribe)

One-way data stream. A publisher node sends messages continuously; any number of subscriber nodes receive them independently. Neither side knows about the other — they only know the topic name.

When to use: Continuous sensor data. IMU readings at 100 Hz, GPS at 5 Hz, battery at 1 Hz — all topics.

/mavros/imu/data is a topic. MAVROS2 publishes; your node subscribes.

Service (Request / Response)

Two-way synchronous call. A client sends a request and blocks until the server responds. Like a function call across the network.

When to use: One-shot queries that need a definite answer. "What is the current system health?" — ask once, get an answer.

/mavros/cmd/arming is a service. You call it to arm the drone.

Action (Goal / Feedback / Result)

Like a service, but for long-running tasks that take seconds or minutes. The client sends a goal; the server sends periodic feedback while executing; finally sends a result. The client can also cancel mid-execution.

When to use: "Fly to this waypoint" — takes 30 seconds, you want distance-remaining feedback, you want to be able to abort.

Parameter

Named configuration values attached to a node. Can be set at launch time or changed at runtime without restarting the node. Not a communication pattern between nodes — a configuration mechanism for a single node.

When to use: Anything you'd put in a config file: waypoint acceptance radius, PID gains, topic names.

ros2 param set /my_node max_altitude 120.0

Key ROS2 Terminology

TermWhat It IsUAV Example
Node A single executable process. One node per responsibility. telemetry_listener, mavros
Message (msg) Typed data structure sent over a topic. Defined in .msg files. sensor_msgs/Imu, geometry_msgs/PoseStamped
Service type (srv) Request + Response pair defined in a .srv file. mavros_msgs/CommandBool (arm/disarm)
Action type (action) Goal + Feedback + Result triple defined in a .action file. Your custom FlyToWaypoint.action
Package A directory containing nodes, launch files, and message definitions. The unit of compilation. aeroknow_bringup, mavros
Workspace A directory containing one or more packages. Built with colcon. ~/aeroknow_ws/
Executor The mechanism that schedules callbacks. rclpy.spin() uses a single-threaded executor by default. Needs MultiThreadedExecutor for action servers (blocking callbacks)

The Node Execution Model — Why Callbacks Matter

This is the single most important concept to get right as a beginner. Every ROS2 node works through callbacks: functions you register that ROS2 calls for you when something happens.

The Callback Mental Model Think of your node as a building with multiple doors. You tell ROS2: "when a message arrives at /mavros/imu/data, open door A and run imu_callback." rclpy.spin(node) is the guard who stands at all doors and triggers the right callback whenever something knocks. Your code does not loop — it waits. The callbacks are where your logic lives. If a callback blocks (e.g., a time.sleep(5)), all other callbacks are frozen until it returns. This is why long-running tasks need Actions and MultiThreadedExecutor.

STEP 02 Create the custom action type: FlyToWaypoint
Why this step exists

ROS2 does not ship with a "fly to waypoint" action type — you define it yourself. An .action file has three sections separated by ---: the Goal (what you're asking the server to do), the Result (what it sends back when done), and the Feedback (what it sends periodically while doing it). This is different from a service (.srv) which only has Request and Response — no feedback.

Step 04: The Telemetry Listener Node

Your first real UAV node. It subscribes to five MAVROS2 topics simultaneously, formats the data, and prints a live summary to the terminal. This pattern — subscribing to multiple sensor topics and acting on their combined data — is the foundation of every subsequent node you'll write.

STEP 04 Write telemetry_listener.py
Why this code is structured this way

We use a class (not bare functions) because the callbacks need to share state — the last GPS reading is needed to display alongside the current battery reading. A class bundles all the subscriptions and their shared state in one place. The timer_callback runs at 2 Hz and prints a consolidated summary — reading 5 sensors at their native rates (100 Hz IMU, 5 Hz GPS, etc.) and printing every callback would flood the terminal. The timer decouples display rate from sensor rate.

bash 
$ nano ~/aeroknow_ws/src/aeroknow_bringup/aeroknow_bringup/telemetry_listener.py
python · telemetry_listener.py 
#!/usr/bin/env python3
"""
telemetry_listener.py
─────────────────────
Subscribes to five MAVROS2 topics and prints a live telemetry
summary at 2 Hz. Demonstrates ROS2 pub/sub with multiple
simultaneous subscriptions sharing state via class members.

Topics subscribed:
  /mavros/imu/data               → sensor_msgs/Imu
  /mavros/global_position/global → sensor_msgs/NavSatFix
  /mavros/global_position/rel_alt→ std_msgs/Float64
  /mavros/state                  → mavros_msgs/State
  /mavros/battery                → sensor_msgs/BatteryState
"""

import rclpy
from rclpy.node import Node
from rclpy.qos import QoSProfile, ReliabilityPolicy, HistoryPolicy

from sensor_msgs.msg import Imu, NavSatFix, BatteryState
from mavros_msgs.msg import State
from std_msgs.msg   import Float64

import math


class TelemetryListener(Node):

    def __init__(self):
        # Every node has a unique name — used in rqt_graph and for logging
        super().__init__('telemetry_listener')

        # ── Shared state ────────────────────────────────────────────
        # These are updated by callbacks and read by the display timer.
        # None means "not yet received" — we display --- until data arrives.
        self.lat        = None
        self.lon        = None
        self.alt_rel    = None   # altitude above home [m]
        self.roll_deg   = None
        self.pitch_deg  = None
        self.battery_v  = None
        self.battery_pct= None
        self.mode       = '—'
        self.armed      = False
        self.connected  = False

        # ── QoS profiles ────────────────────────────────────────────
        # MAVROS2 sensor topics use BEST_EFFORT (fastest, may drop packets).
        # State topics use RELIABLE (guaranteed delivery, slightly slower).
        # Using the wrong profile causes "no subscribers / no publishers" warnings
        # even though the topic exists. This is the most common L02 mistake.
        sensor_qos = QoSProfile(
            reliability=ReliabilityPolicy.BEST_EFFORT,
            history=HistoryPolicy.KEEP_LAST,
            depth=1,
        )
        reliable_qos = QoSProfile(
            reliability=ReliabilityPolicy.RELIABLE,
            history=HistoryPolicy.KEEP_LAST,
            depth=10,
        )

        # ── Subscriptions ───────────────────────────────────────────
        # Each create_subscription call registers a callback.
        # ROS2 calls the callback whenever a message arrives on that topic.
        # The '1' / '10' at the end is the queue depth — how many unprocessed
        # messages to buffer before dropping. For display purposes, 1 is fine.
        self.create_subscription(Imu, '/mavros/imu/data',
                                 self.imu_callback, sensor_qos)

        self.create_subscription(NavSatFix, '/mavros/global_position/global',
                                 self.gps_callback, sensor_qos)

        self.create_subscription(Float64, '/mavros/global_position/rel_alt',
                                 self.alt_callback, sensor_qos)

        self.create_subscription(State, '/mavros/state',
                                 self.state_callback, reliable_qos)

        self.create_subscription(BatteryState, '/mavros/battery',
                                 self.battery_callback, sensor_qos)

        # ── Display timer ───────────────────────────────────────────
        # Fires self.display_callback every 0.5 seconds (2 Hz).
        # Completely independent of the sensor callbacks — decouples
        # display rate from sensor rate. IMU comes in at 100 Hz;
        # we only print 2 times per second.
        self.create_timer(0.5, self.display_callback)

        self.get_logger().info('TelemetryListener started — waiting for MAVROS2 data...')

    # ── Callbacks ───────────────────────────────────────────────────

    def imu_callback(self, msg: Imu):
        # Convert quaternion → Euler angles (roll, pitch only — yaw needs magnetometer)
        # Formula: roll = atan2(2*(qw*qx + qy*qz), 1 - 2*(qx² + qy²))
        q = msg.orientation
        sinr_cosp = 2.0 * (q.w * q.x + q.y * q.z)
        cosr_cosp = 1.0 - 2.0 * (q.x * q.x + q.y * q.y)
        self.roll_deg = math.degrees(math.atan2(sinr_cosp, cosr_cosp))

        sinp = 2.0 * (q.w * q.y - q.z * q.x)
        sinp = max(-1.0, min(1.0, sinp))   # clamp to [-1, 1] for asin
        self.pitch_deg = math.degrees(math.asin(sinp))

    def gps_callback(self, msg: NavSatFix):
        self.lat = msg.latitude
        self.lon = msg.longitude

    def alt_callback(self, msg: Float64):
        self.alt_rel = msg.data

    def state_callback(self, msg: State):
        self.connected = msg.connected
        self.armed     = msg.armed
        self.mode      = msg.mode

    def battery_callback(self, msg: BatteryState):
        self.battery_v   = msg.voltage
        self.battery_pct = msg.percentage * 100.0   # MAVROS2 sends 0.0–1.0

    def display_callback(self):
        # Format values — show "---" if data not yet received
        def fmt(v, fmt_str='{:.2f}'):
            return fmt_str.format(v) if v is not None else '---'

        arm_str  = '\033[92mARMED  \033[0m' if self.armed else '\033[90mDISARMED\033[0m'
        conn_str = '\033[92m●\033[0m'         if self.connected else '\033[91m●\033[0m'

        print(f'\033[2J\033[H', end='')   # clear terminal
        print(f'┌─────────────────────────────────────────┐')
        print(f'│  AEROKNOW · TELEMETRY LISTENER · L02   │')
        print(f'├─────────────────────────────────────────┤')
        print(f'│  FCU  {conn_str} {self.mode:<16} {arm_str} │')
        print(f'├─────────────────────────────────────────┤')
        print(f'│  GPS   lat: {fmt(self.lat, "{:>10.6f}")}  lon: {fmt(self.lon, "{:>11.6f}")} │')
        print(f'│  ALT   {fmt(self.alt_rel, "{:>7.2f}")} m above home               │')
        print(f'├─────────────────────────────────────────┤')
        print(f'│  ROLL  {fmt(self.roll_deg,  "{:>+7.2f}")} deg                        │')
        print(f'│  PITCH {fmt(self.pitch_deg, "{:>+7.2f}")} deg                        │')
        print(f'├─────────────────────────────────────────┤')
        print(f'│  BATT  {fmt(self.battery_v, "{:>5.2f}")} V  {fmt(self.battery_pct, "{:>5.1f}")} %              │')
        print(f'└─────────────────────────────────────────┘')


def main(args=None):
    rclpy.init(args=args)
    node = TelemetryListener()
    try:
        rclpy.spin(node)     # blocks here, running callbacks until Ctrl+C
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()

Step 05: The Health Check Service

A service is different from a topic: nothing happens until a client explicitly asks. The health service listens for requests and responds with the current system health status. The GCS in L07 will call this service to populate its health panel.

STEP 05 Write heartbeat_service.py
Why this code is structured this way

This node does two things simultaneously: it subscribes to sensor topics (to track their health) and it serves requests (to report that health). These happen on different threads — a MultiThreadedExecutor is used so the service can respond even while subscription callbacks are running. Without it, a blocking service response would prevent sensor callbacks from firing and the health data would go stale while the service was occupied.

bash 
$ nano ~/aeroknow_ws/src/aeroknow_bringup/aeroknow_bringup/heartbeat_service.py
python · heartbeat_service.py 
#!/usr/bin/env python3
"""
heartbeat_service.py
────────────────────
Service server that returns system health on request.
Uses the built-in std_srvs/Trigger service type:
  Request:  (empty)
  Response: bool success, string message

Called by: GCS in L07, pre-flight checks in L03.
"""

import rclpy
from rclpy.node          import Node
from rclpy.executors     import MultiThreadedExecutor
from rclpy.callback_group import ReentrantCallbackGroup
from rclpy.qos           import QoSProfile, ReliabilityPolicy, HistoryPolicy

from std_srvs.srv   import Trigger
from mavros_msgs.msg import State
from sensor_msgs.msg import NavSatFix, BatteryState

import time


class HeartbeatService(Node):

    def __init__(self):
        super().__init__('heartbeat_service')

        # ReentrantCallbackGroup allows multiple callbacks to run concurrently.
        # Without this, the service callback would block all subscriptions.
        cb_group = ReentrantCallbackGroup()

        sensor_qos = QoSProfile(
            reliability=ReliabilityPolicy.BEST_EFFORT,
            history=HistoryPolicy.KEEP_LAST, depth=1
        )
        reliable_qos = QoSProfile(
            reliability=ReliabilityPolicy.RELIABLE,
            history=HistoryPolicy.KEEP_LAST, depth=10
        )

        # ── Health tracking state ────────────────────────────────────
        self.fcu_connected = False
        self.armed         = False
        self.mode          = 'UNKNOWN'
        self.gps_fix       = 0   # NavSatFix.status: 0=no fix, 1=fix, 2=SBAS
        self.battery_pct   = 0.0
        self.last_imu_t    = 0.0   # epoch time of last IMU message

        # ── Subscriptions (monitoring sensor health) ─────────────────
        self.create_subscription(State, '/mavros/state',
            self.state_cb, reliable_qos, callback_group=cb_group)

        self.create_subscription(NavSatFix, '/mavros/global_position/global',
            self.gps_cb, sensor_qos, callback_group=cb_group)

        self.create_subscription(BatteryState, '/mavros/battery',
            self.battery_cb, sensor_qos, callback_group=cb_group)

        # Track IMU freshness: subscribe but only record timestamp
        from sensor_msgs.msg import Imu
        self.create_subscription(Imu, '/mavros/imu/data',
            lambda _: setattr(self, 'last_imu_t', time.time()),
            sensor_qos, callback_group=cb_group)

        # ── Service server ───────────────────────────────────────────
        # 'get_system_health' is the service name; clients call it with:
        #   ros2 service call /get_system_health std_srvs/srv/Trigger
        # or programmatically from the GCS in L07.
        self.create_service(Trigger, 'get_system_health',
            self.health_callback, callback_group=cb_group)

        self.get_logger().info('HeartbeatService ready at /get_system_health')

    # ── Subscription callbacks ───────────────────────────────────────

    def state_cb(self, msg: State):
        self.fcu_connected = msg.connected
        self.armed         = msg.armed
        self.mode          = msg.mode

    def gps_cb(self, msg: NavSatFix):
        self.gps_fix = msg.status.status   # -1=no fix, 0=fix, 1=SBAS fix

    def battery_cb(self, msg: BatteryState):
        self.battery_pct = msg.percentage * 100.0

    # ── Service callback ─────────────────────────────────────────────

    def health_callback(self, request, response):
        """
        Called each time a client calls /get_system_health.
        Builds a human-readable health report and returns it.
        The Trigger service type is the simplest possible:
        - request:  empty (client just knocks on the door)
        - response: bool success + string message
        """
        issues = []

        if not self.fcu_connected:
            issues.append('FCU not connected')
        if self.gps_fix < 0:
            issues.append('No GPS fix')
        if self.battery_pct < 20.0:
            issues.append(f'Low battery: {self.battery_pct:.1f}%')
        if time.time() - self.last_imu_t > 1.0:
            issues.append('IMU stale (>1s since last message)')

        if issues:
            response.success = False
            response.message = 'DEGRADED | ' + ' | '.join(issues)
        else:
            response.success = True
            response.message = (
                f'OK | FCU connected | GPS fix={self.gps_fix} | '
                f'Battery {self.battery_pct:.1f}% | '
                f'Mode {self.mode} | Armed={self.armed}'
            )

        # Log every request so we have an audit trail
        self.get_logger().info(f'Health request → {response.message}')
        return response


def main(args=None):
    rclpy.init(args=args)
    node = HeartbeatService()

    # MultiThreadedExecutor: allows service + subscriptions to run concurrently
    executor = MultiThreadedExecutor(num_threads=4)
    executor.add_node(node)
    try:
        executor.spin()
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()

Step 06: The Mission Action Server

Actions are the right tool for anything that takes more than a fraction of a second to complete. "Fly to this waypoint" is the canonical UAV action — it takes 10–60 seconds, you want progress feedback while it's running, and you want to be able to cancel it if the situation changes.

STEP 06 Write mission_action_server.py
Why this code is structured this way

The action server has three callbacks instead of one: goal_callback (should I accept this goal?), cancel_callback (should I accept a cancellation?), and execute_callback (actually do the work). The execute callback is the long-running one — it loops, publishes setpoints to MAVROS2, checks distance, sends feedback, and eventually returns a result. We must use MultiThreadedExecutor here too because execute_callback blocks while it's running the mission, and we still need the subscription callbacks (to read position) to fire concurrently.

bash 
$ nano ~/aeroknow_ws/src/aeroknow_bringup/aeroknow_bringup/mission_action_server.py
python · mission_action_server.py 
#!/usr/bin/env python3
"""
mission_action_server.py
────────────────────────
Action server that flies the drone to a GPS waypoint.
Action type: FlyToWaypoint (defined in action/FlyToWaypoint.action)

Goal:     target_lat, target_lon, target_alt, acceptance_radius
Feedback: distance_to_wp, estimated_time_s, phase
Result:   success, message, final_distance
"""

import rclpy
from rclpy.node          import Node
from rclpy.action        import ActionServer, CancelResponse, GoalResponse
from rclpy.executors     import MultiThreadedExecutor
from rclpy.callback_group import ReentrantCallbackGroup
from rclpy.qos           import QoSProfile, ReliabilityPolicy, HistoryPolicy

from mavros_msgs.msg import State
from sensor_msgs.msg import NavSatFix
from geographic_msgs.msg import GeoPoseStamped

# Note: this import works after colcon build generates the Python bindings
# from our FlyToWaypoint.action definition file.
from aeroknow_bringup.action import FlyToWaypoint

import math
import time


def haversine_m(lat1, lon1, lat2, lon2) -> float:
    """Haversine formula — great-circle distance in metres."""
    R = 6_371_000   # Earth radius [m]
    φ1, φ2 = math.radians(lat1), math.radians(lat2)
    Δφ = math.radians(lat2 - lat1)
    Δλ = math.radians(lon2 - lon1)
    a = (math.sin(Δφ/2)**2
         + math.cos(φ1) * math.cos(φ2) * math.sin(Δλ/2)**2)
    return R * 2 * math.atan2(math.sqrt(a), math.sqrt(1-a))


class MissionActionServer(Node):

    def __init__(self):
        super().__init__('mission_action_server')

        cb_group = ReentrantCallbackGroup()

        sensor_qos = QoSProfile(
            reliability=ReliabilityPolicy.BEST_EFFORT,
            history=HistoryPolicy.KEEP_LAST, depth=1
        )

        # ── State tracking ───────────────────────────────────────────
        self.current_lat  = None
        self.current_lon  = None
        self.current_alt  = None
        self.armed        = False
        self.mode         = 'UNKNOWN'

        self.create_subscription(NavSatFix, '/mavros/global_position/global',
            self.gps_cb, sensor_qos, callback_group=cb_group)

        self.create_subscription(State, '/mavros/state',
            self.state_cb,
            QoSProfile(reliability=ReliabilityPolicy.RELIABLE,
                       history=HistoryPolicy.KEEP_LAST, depth=10),
            callback_group=cb_group)

        # ── Publisher: send global waypoint setpoints to MAVROS2 ──────
        # /mavros/setpoint_position/global sends the drone to a GPS coordinate
        # in AUTO.MISSION-compatible format. MAVROS2 handles the rest.
        self.wp_pub = self.create_publisher(
            GeoPoseStamped,
            '/mavros/setpoint_position/global',
            10
        )

        # ── Action server ────────────────────────────────────────────
        # Three callbacks:
        #   goal_callback:   called when a client sends a goal — accept or reject
        #   cancel_callback: called if client cancels — accept or reject cancel
        #   execute_callback: the actual long-running task
        self._action_server = ActionServer(
            self,
            FlyToWaypoint,
            'fly_to_waypoint',
            goal_callback=self.goal_callback,
            cancel_callback=self.cancel_callback,
            execute_callback=self.execute_callback,
            callback_group=cb_group,
        )

        self.get_logger().info('MissionActionServer ready at /fly_to_waypoint')

    # ── Subscription callbacks ───────────────────────────────────────

    def gps_cb(self, msg: NavSatFix):
        self.current_lat = msg.latitude
        self.current_lon = msg.longitude
        self.current_alt = msg.altitude

    def state_cb(self, msg: State):
        self.armed = msg.armed
        self.mode  = msg.mode

    # ── Action callbacks ─────────────────────────────────────────────

    def goal_callback(self, goal_request):
        """Decide whether to accept an incoming goal."""
        if not self.armed:
            self.get_logger().warn('Goal rejected: drone not armed')
            return GoalResponse.REJECT
        if self.current_lat is None:
            self.get_logger().warn('Goal rejected: no GPS fix yet')
            return GoalResponse.REJECT
        self.get_logger().info(
            f'Goal accepted: fly to ({goal_request.target_lat:.6f}, '
            f'{goal_request.target_lon:.6f}) at {goal_request.target_alt:.1f}m'
        )
        return GoalResponse.ACCEPT

    def cancel_callback(self, goal_handle):
        """Always accept cancellation — safe design principle."""
        self.get_logger().info('Cancel request received — accepting')
        return CancelResponse.ACCEPT

    def execute_callback(self, goal_handle):
        """
        The main execution loop. Runs until:
        - Drone arrives within acceptance_radius → SUCCESS
        - Client cancels → CANCELED
        - Timeout (120s) → ABORTED
        Sends feedback every 0.5 s.
        """
        goal    = goal_handle.request
        feedback= FlyToWaypoint.Feedback()
        result  = FlyToWaypoint.Result()

        RATE_HZ = 2.0   # feedback + setpoint rate
        TIMEOUT = 120.0 # max seconds to complete
        t_start = time.time()

        self.get_logger().info(f'Executing: fly to ({goal.target_lat:.6f}, {goal.target_lon:.6f})')

        while rclpy.ok():
            # ── Check cancellation ──────────────────────────────────
            if goal_handle.is_cancel_requested:
                goal_handle.canceled()
                result.success = False
                result.message = 'aborted'
                result.final_distance = 999.0
                return result

            # ── Check timeout ───────────────────────────────────────
            elapsed = time.time() - t_start
            if elapsed > TIMEOUT:
                goal_handle.abort()
                result.success = False
                result.message = 'timeout'
                result.final_distance = haversine_m(
                    self.current_lat, self.current_lon,
                    goal.target_lat, goal.target_lon)
                return result

            # ── Compute distance ────────────────────────────────────
            if self.current_lat is None:
                time.sleep(1.0 / RATE_HZ)
                continue

            dist = haversine_m(
                self.current_lat, self.current_lon,
                goal.target_lat,  goal.target_lon)

            # ── Check arrival ───────────────────────────────────────
            if dist <= goal.acceptance_radius:
                goal_handle.succeed()
                result.success        = True
                result.message        = 'arrived'
                result.final_distance = dist
                self.get_logger().info(f'Waypoint reached. Final distance: {dist:.2f}m')
                return result

            # ── Publish waypoint setpoint ────────────────────────────
            # Send the target GPS coordinate to MAVROS2 setpoint topic.
            # MAVROS2 in OFFBOARD mode will route this to PX4's position controller.
            wp_msg = GeoPoseStamped()
            wp_msg.header.stamp = self.get_clock().now().to_msg()
            wp_msg.pose.position.latitude  = goal.target_lat
            wp_msg.pose.position.longitude = goal.target_lon
            wp_msg.pose.position.altitude  = goal.target_alt
            self.wp_pub.publish(wp_msg)

            # ── Send feedback ───────────────────────────────────────
            speed_mps = 8.0   # rough cruise speed for ETA estimate
            alt_diff  = abs((self.current_alt or goal.target_alt) - goal.target_alt)

            feedback.distance_to_wp   = dist
            feedback.estimated_time_s = (dist / speed_mps) + (alt_diff / 2.0)
            feedback.phase = (
                'climbing'   if alt_diff > 5.0 and self.current_alt < goal.target_alt
                else 'descending' if alt_diff > 5.0
                else 'en_route'
            )
            goal_handle.publish_feedback(feedback)

            time.sleep(1.0 / RATE_HZ)


def main(args=None):
    rclpy.init(args=args)
    node = MissionActionServer()
    executor = MultiThreadedExecutor(num_threads=4)
    executor.add_node(node)
    try:
        executor.spin()
    except KeyboardInterrupt:
        pass
    finally:
        node.destroy_node()
        rclpy.shutdown()


if __name__ == '__main__':
    main()

Step 07: The Launch File

Starting three nodes manually in three terminals is fine for development. A launch file starts all of them with one command, sets parameters, and handles dependencies between them. From L03 onwards, you will always start the full stack via a single launch command.

STEP 07 Write bringup_launch.py
Why this code is structured this way

ROS2 launch files are Python scripts — not XML (that was ROS1). They define a list of Node and DeclareLaunchArgument objects returned from generate_launch_description(). The LaunchDescription is the recipe; ros2 launch is the chef that executes it. Parameters are declared as launch arguments so they can be overridden from the command line without editing the file — critical for switching between simulation and real hardware.

bash 
$ nano ~/aeroknow_ws/src/aeroknow_bringup/launch/bringup_launch.py
python · bringup_launch.py 
#!/usr/bin/env python3
"""
bringup_launch.py
─────────────────
Launches all three L02 nodes in a single command.
Usage:
  ros2 launch aeroknow_bringup bringup_launch.py
  ros2 launch aeroknow_bringup bringup_launch.py vehicle_id:=2
"""

from launch import LaunchDescription
from launch.actions import DeclareLaunchArgument, LogInfo
from launch.substitutions import LaunchConfiguration
from launch_ros.actions import Node


def generate_launch_description():
    # ── Declare configurable arguments ──────────────────────────────
    # These can be overridden on the command line:
    #   ros2 launch ... vehicle_id:=2
    # Without override, the default values below are used.
    vehicle_id_arg = DeclareLaunchArgument(
        'vehicle_id',
        default_value='1',
        description='Vehicle ID for multi-drone namespacing (L08)'
    )

    mavros_ns_arg = DeclareLaunchArgument(
        'mavros_ns',
        default_value='',
        description='MAVROS2 namespace prefix (empty for single drone)'
    )

    # ── Telemetry listener node ──────────────────────────────────────
    telemetry_node = Node(
        package='aeroknow_bringup',
        executable='telemetry_listener',
        name='telemetry_listener',
        output='screen',
        parameters=[{
            'display_rate_hz': 2.0,
            'vehicle_id': LaunchConfiguration('vehicle_id'),
        }],
        remappings=[
            # Remappings allow the same node to work in different namespaces.
            # In L08, these will be prefixed with /drone_1/, /drone_2/, etc.
        ],
    )

    # ── Heartbeat service node ───────────────────────────────────────
    heartbeat_node = Node(
        package='aeroknow_bringup',
        executable='heartbeat_service',
        name='heartbeat_service',
        output='log',   # log output only — no terminal spam
        parameters=[{
            'battery_warn_pct': 20.0,
            'imu_stale_sec': 1.0,
        }],
    )

    # ── Mission action server node ───────────────────────────────────
    mission_node = Node(
        package='aeroknow_bringup',
        executable='mission_action_server',
        name='mission_action_server',
        output='screen',
        parameters=[{
            'cruise_speed_mps': 8.0,
            'mission_timeout_s': 120.0,
        }],
    )

    return LaunchDescription([
        # Arguments first
        vehicle_id_arg,
        mavros_ns_arg,
        # Info log so launch output is clear
        LogInfo(msg='[AEROKNOW] Starting bringup: telemetry + health + mission nodes'),
        # Nodes
        telemetry_node,
        heartbeat_node,
        mission_node,
    ])

Step 08: QoS Profiles — Why They Matter for UAV Code

Quality of Service (QoS) profiles are the most overlooked source of silent failures in ROS2 UAV code. If a publisher and subscriber have incompatible QoS profiles, no error is thrown — the subscriber simply never receives any messages. The node runs, the topic exists, but the data never flows. This looks identical to "the sensor is not publishing."

The Two Profiles You Must Know

ProfileReliabilityWhen to UseMAVROS2 Topics
BEST_EFFORT May drop messages. Fast. Lower CPU. High-rate sensor data where a missed sample is fine (IMU at 100 Hz — missing one is harmless) /mavros/imu/data, /mavros/battery, /mavros/global_position/global
RELIABLE Guaranteed delivery. Retransmits on loss. Low-rate, safety-critical data where missing a message is dangerous (mode changes, arm commands) /mavros/state, /mavros/set_mode, /mavros/cmd/arming

The rule: your subscriber QoS must match or be less strict than the publisher QoS. MAVROS2 publishes sensor topics with BEST_EFFORT. If you subscribe with RELIABLE, ROS2 sees an incompatible pair and connects nothing — silently. You can check what profile a topic is actually using:

bash 
$ ros2 topic info /mavros/imu/data --verbose
Output — note the QoS settings

Node name: /mavros
  Reliability: BEST_EFFORT   <-- must match this in your subscriber
  Durability: VOLATILE
  Deadline: infinite
  Lifespan: infinite
  Liveliness: AUTOMATIC
  Liveliness lease: infinite

Subscription count: 1

Common Errors & Fixes

These are the errors that reliably appear in L02 — documented here so you spend minutes, not hours, recovering from them.

Error 01 — Node receives no data despite topic existing No messages on /mavros/imu/data (subscriber running but nothing arrives)

Cause: QoS mismatch. Your subscriber profile does not match the publisher. This is a silent failure — no warning, no error, just empty callbacks. Fix: Run ros2 topic info /mavros/imu/data --verbose and match the Reliability field exactly in your subscriber. MAVROS2 sensor topics are BEST_EFFORT; state topics are RELIABLE. Copy the QoS definitions from Step 04 exactly as written.

Error 02 — ModuleNotFoundError: aeroknow_bringup.action ModuleNotFoundError: No module named 'aeroknow_bringup.action'

Cause: The action Python bindings were not generated because the package wasn't rebuilt after adding the action/ directory and the rosidl buildtool dependency. Fix: (1) Confirm package.xml has the rosidl_default_generators buildtool dependency. (2) Run colcon build --symlink-install --packages-select aeroknow_bringup from ~/aeroknow_ws. (3) Run source install/setup.bash in the same terminal before running the node.

Error 03 — Action goal rejected immediately Goal rejected: drone not armed

Cause: The goal_callback in mission_action_server.py checks that the drone is armed before accepting. In SITL, the drone starts disarmed. Fix: Arm the drone first from the PX4 shell: type commander arm in Terminal 1 (the PX4 shell). Then re-send the action goal. In L03 you will write proper arm/takeoff logic that handles this automatically.

Error 04 — colcon build error: "No package 'geographic_msgs'" ModuleNotFoundError: No module named 'geographic_msgs'

Cause: geographic_msgs is not installed by default with ROS2 Humble. Fix: sudo apt install ros-humble-geographic-msgs, then rebuild.

Error 05 — Launch file exits immediately with no error [launch]: process has finished cleanly (no output)

Cause: A syntax error in bringup_launch.py causes the launch system to fail silently. Python launch files are not compiled — errors only appear at run time. Fix: Test the launch file directly with Python to see the traceback: python3 ~/aeroknow_ws/src/aeroknow_bringup/launch/bringup_launch.py. The real error will be printed, usually a missing import or indentation issue.

Extension Challenges

You have four ROS2 components running. Here are three harder things to try before L03.

  1. Add a velocity subscriber to telemetry_listener. Subscribe to /mavros/local_position/velocity_local (type geometry_msgs/TwistStamped) and add a speed display row to the terminal output. The topic name is BEST_EFFORT. Use math.sqrt(vx² + vy²) for horizontal groundspeed. This is the first data the GCS in L07 needs beyond what you already have.
  2. Extend heartbeat_service with a latency check. Add a fourth health criterion: the IMU must be publishing at ≥ 50 Hz. Store the last 10 IMU timestamps and compute the rolling average rate. Report "IMU degraded (rate: X Hz)" if it drops below 50 Hz. This matters in L05 when the EKF depends on consistent IMU timing.
  3. Write a mission action client. Create a new script mission_client.py that programmatically sends a goal to /fly_to_waypoint, prints feedback messages, and prints the final result. Use rclpy.action.ActionClient. This exact client pattern is used in L03 to chain multiple waypoints into a full mission. Hint: look up send_goal_async and how to attach a feedback_callback to it.
