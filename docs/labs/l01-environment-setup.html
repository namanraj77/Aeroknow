Environment Setup & First Flight

Install the complete AEROKNOW simulation stack on Ubuntu 22.04 — ROS2 Humble, PX4 v1.14 SITL, Gazebo Garden, and MAVROS2. By the end of this lab, a simulated drone is hovering in Gazebo and every topic in the stack is verified alive. This is the foundation everything else depends on.

⏱ ~2–3 hours wall clock
📋 9 steps
💻 Ubuntu 22.04 required (bare metal or VM)
🔧 No hardware required
ROS2 Humble PX4 v1.14 Gazebo Garden

What You Are Building

By the end of L01, this is the complete system running on your machine. Every box is a separate process. Every arrow is a communication channel. Later labs add nodes on top of this stack — nothing here gets replaced.

Theory Link

This lab implements the C2 architecture described in Module 01 — UAV Systems Overview. The GCS → MAVROS2 → PX4 chain you are installing is exactly the command-and-control pipeline covered in §1.4. If the acronyms feel unfamiliar, read M01 §1.1–1.4 first.

Step 1 — Prerequisites

Before installing anything, verify your machine meets every requirement below. Skipping this check is the single most common reason a setup fails three hours in.

▸

Ubuntu 22.04 LTS (Jammy) — bare metal or VMware/VirtualBox with hardware virtualisation enabled. WSL2 is not supported — Gazebo requires a real GPU/display stack. Check: lsb_release -a must show 22.04.

▸

8 GB RAM minimum, 16 GB recommended. PX4 SITL + Gazebo Garden + ROS2 together consume ~4–6 GB at runtime. Check: free -h.

▸

30 GB free disk space. PX4 source tree is ~3 GB, Gazebo models cache grows over time. Check: df -h ~.

▸

Internet connection. All packages are pulled from apt, pip, and GitHub. A slow connection adds time but not failure risk — the installs are resumable.

▸

sudo access. Several steps require root. Check: sudo whoami must print root.

VM note If running in VirtualBox: set Video Memory to 128 MB, enable 3D Acceleration, and allocate at least 4 CPU cores. Gazebo Garden will refuse to launch with less than 256 MB video memory if software rendering fallback is disabled.

Step 2 — Install ROS2 Humble

STEP 02.1 Add the ROS2 apt repository
Why

ROS2 Humble is not in Ubuntu's default apt sources. We add the official OSRF repository and its GPG key so apt can find and verify every ROS2 package. This is a one-time setup — once the source is added, all future apt install ros-humble-* commands work without repeating this.

bash 
$ sudo apt update && sudo apt install -y curl gnupg lsb-release
$ sudo curl -sSL https://raw.githubusercontent.com/ros/rosdistro/master/ros.key \
    -o /usr/share/keyrings/ros-archive-keyring.gpg
$ echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/ros-archive-keyring.gpg] \
    http://packages.ros.org/ros2/ubuntu $(. /etc/os-release && echo $UBUNTU_CODENAME) main" \
    | sudo tee /etc/apt/sources.list.d/ros2.list > /dev/null
$ sudo apt update
STEP 02.2 Install ROS2 Humble Desktop + colcon
Why

ros-humble-desktop is the full install — it includes rviz2, rqt, and all core libraries. We need the desktop variant (not base) because L02 uses rqt_graph to visualise the node network. colcon is the ROS2 build system that compiles our aeroknow_ws workspace in later labs.

bash 
$ sudo apt install -y ros-humble-desktop python3-colcon-common-extensions
$ echo "source /opt/ros/humble/setup.bash" >> ~/.bashrc
$ source ~/.bashrc
Expected output 
Reading package lists... Done
Building dependency tree... Done
Setting up ros-humble-desktop (0.10.0-1jammy.20240101) ...
Setting up python3-colcon-common-extensions ...

Verify the install worked:

bash 
$ ros2 --version
Expected output 
ros2, version 0.18.x

Step 3 — Install PX4 Autopilot v1.14 SITL

SITL stands for Software In The Loop. PX4 compiles to a native Linux binary that runs the exact same flight control code as real hardware — but instead of reading physical IMU sensors, it reads simulated sensor data from Gazebo. The MAVLink UDP ports it opens are identical to a real vehicle.

STEP 03.1 Clone and run the PX4 setup script
Why v1.14 specifically

PX4 v1.14 is the latest LTS-equivalent release that is known to work with Gazebo Garden and the ROS2 Humble MAVROS2 bridge. v1.15 introduced breaking changes to the uXRCE-DDS interface that are not yet stable. We version-lock everything in AEROKNOW Labs to eliminate setup divergence.

bash 
$ cd ~
$ git clone https://github.com/PX4/PX4-Autopilot.git --recursive --branch v1.14.0
$ cd PX4-Autopilot
$ bash ./Tools/setup/ubuntu.sh
Time warning The ubuntu.sh script installs ~200 packages including GCC arm cross-compiler toolchains, Python build tools, and Gazebo dependencies. On a typical broadband connection this takes 15–25 minutes. Do not interrupt it.
Expected final lines of ubuntu.sh 
-- Installing: /usr/local/bin/px4-alias.sh
PX4 development environment setup complete.
Please log out and back in, or run: source ~/.bashrc
STEP 03.2 Build PX4 SITL
Why

This compiles PX4 for the gz_x500 target — a simulated X500 quadrotor with Gazebo Garden as the physics backend. The first build takes 5–10 minutes and produces the px4 binary in build/px4_sitl_default/. Subsequent builds are incremental and fast.

bash 
$ cd ~/PX4-Autopilot
$ make px4_sitl gz_x500
Expected final lines — Gazebo should open automatically 
INFO  [px4] Startup script returned successfully
INFO  [commander] Ready for takeoff!
pxh>

Gazebo Garden will launch and show the X500 quadrotor sitting on the ground. You now have a working PX4 SITL instance. Close Gazebo and the PX4 shell with Ctrl+C — we will launch them properly in Step 8.

Step 4 — Verify Gazebo Garden

STEP 04.1 Install the ROS2–Gazebo bridge
Why

Gazebo Garden uses a completely separate topic system from ROS2. The ros_gz_bridge package translates between them — a Gazebo camera image becomes a ROS2 sensor_msgs/Image, a Gazebo IMU becomes a ROS2 sensor_msgs/Imu. Without the bridge, your ROS2 nodes cannot see any sensor data from the simulation.

bash 
$ sudo apt install -y ros-humble-ros-gz-bridge ros-humble-ros-gz-sim \
    ros-humble-ros-gz-interfaces gz-garden

Verify Gazebo is accessible from the command line:

bash 
$ gz sim --version
Expected output 
Gazebo Sim, version 7.x.x

Step 5 — Install MAVROS2

STEP 05.1 Install MAVROS2 and its GeographicLib datasets
Why MAVROS2 exists

MAVLink is the binary protocol PX4 uses to communicate — it is compact and efficient but not natively readable by ROS2 nodes. MAVROS2 is a bridge: it speaks MAVLink to PX4 over UDP and re-publishes everything as standard ROS2 topics. Your Python nodes never touch raw MAVLink — they just subscribe to /mavros/imu/data and publish to /mavros/setpoint_position/local.

bash 
$ sudo apt install -y ros-humble-mavros ros-humble-mavros-extras
$ ros2 run mavros install_geographiclib_datasets.sh
GeographicLib is mandatory The install_geographiclib_datasets.sh script downloads ~65 MB of magnetic field and geoid model data. MAVROS2 will crash at startup without it with a cryptic "geoid file not found" error. Always run this after installing mavros.
Expected output 
Downloading geoid dataset (egm96-5)...
Done: /usr/share/GeographicLib/geoids/egm96-5.pgm
Downloading gravity dataset (egm96)...
Done: /usr/share/GeographicLib/gravity/egm96.egm

All lab code lives in one unified ROS2 workspace — aeroknow_ws/. You will add new packages to this workspace in every subsequent lab. Creating it now means L02 can start immediately.

Step 8 — First Flight

This is the payoff step. We launch PX4 SITL with Gazebo, connect MAVROS2, and command a takeoff — all from separate terminals. You need three terminal windows open simultaneously.

STEP 08.1 Terminal 1 — Launch PX4 SITL + Gazebo
bash · Terminal 1 
$ cd ~/PX4-Autopilot
$ make px4_sitl gz_x500

Wait until you see INFO [commander] Ready for takeoff! and the Gazebo window shows the drone.

STEP 08.2 Terminal 2 — Launch MAVROS2
Why

MAVROS2 connects to PX4 on UDP port 14550 (the GCS port) and begins translating MAVLink messages into ROS2 topics. The fcu_url parameter tells it where PX4 is. The gcs_url parameter would forward telemetry to a GCS like QGroundControl — we leave it empty for now.

bash · Terminal 2 
$ source /opt/ros/humble/setup.bash
$ ros2 launch mavros px4.launch fcu_url:="udp://:14550@127.0.0.1:14557"
Expected output — MAVROS2 connected 
[INFO] [mavros_node]: FCU URL: udp://:14550@127.0.0.1:14557
[INFO] [mavros_node]: Connected to vehicle
[INFO] [mavros_node]: GCS bridge started
[INFO] [mavros_node]: Mode: MANUAL
STEP 08.3 Terminal 3 — Command takeoff via PX4 shell
Why the PX4 shell for first takeoff

The PX4 shell (pxh>) is a direct interface to the flight controller — no ROS2 involved. Commanding takeoff from here proves PX4 and Gazebo are working before we add the MAVROS2 layer. If this doesn't work, the problem is in the simulation, not in our ROS2 code.

Finding the PX4 shell The PX4 shell is in Terminal 1 — the one where you ran make px4_sitl gz_x500. Look for the pxh> prompt at the bottom.
PX4 shell (Terminal 1 · pxh> prompt) 
pxh> commander takeoff
Expected: drone lifts to 2.5m in Gazebo 
INFO  [commander] Takeoff detected
INFO  [navigator] Takeoff completed

The drone should visibly rise in the Gazebo window and hold altitude. This confirms PX4 SITL + Gazebo Garden is fully functional. 🎉

Step 9 — Verify All Topics (sanity_check.py)

A working Gazebo + PX4 window is not enough. We need to verify that every ROS2 topic we will use in later labs is actually publishing data. This script is the first Python node you build in aeroknow_ws.

STEP 09.1 Write sanity_check.py
What this node does

sanity_check.py subscribes to 5 critical topics simultaneously and waits for at least one message on each within 10 seconds. If any topic is silent, it prints a red error with a diagnostic hint. This catches 90% of broken setups — a MAVROS2 connection issue shows up immediately rather than mysteriously causing failures in L03 flight code.

python 
#!/usr/bin/env python3
# sanity_check.py — L01: verify the full stack is alive
# Save to: ~/aeroknow_ws/src/aeroknow_utils/aeroknow_utils/sanity_check.py

import rclpy
from rclpy.node import Node
from sensor_msgs.msg import Imu, NavSatFix, BatteryState
from mavros_msgs.msg import State
from std_msgs.msg import Float64
import threading, time

# ── Topics to verify ───────────────────────────────────────────────
# Each entry: (topic_name, msg_type, description)
REQUIRED_TOPICS = [
    ('/mavros/imu/data',                  Imu,          'IMU (roll/pitch/yaw rates)'),
    ('/mavros/global_position/global',    NavSatFix,    'GPS fix'),
    ('/mavros/state',                      State,        'Vehicle state (armed, mode)'),
    ('/mavros/battery',                    BatteryState, 'Battery telemetry'),
    ('/mavros/global_position/rel_alt',   Float64,      'Relative altitude'),
]

class SanityCheck(Node):
    def __init__(self):
        super().__init__('sanity_check')
        self.received = {t[0]: False for t in REQUIRED_TOPICS}
        self.subs = []

        # Subscribe to all required topics
        for topic, msg_type, desc in REQUIRED_TOPICS:
            cb = lambda msg, t=topic: self._mark(t)
            self.subs.append(self.create_subscription(msg_type, topic, cb, 10))

        self.get_logger().info('Waiting for topics (10s timeout)...')
        self._timer = self.create_timer(10.0, self._report)

    def _mark(self, topic):
        if not self.received[topic]:
            self.received[topic] = True
            self.get_logger().info(f'  ✓  {topic}')

    def _report(self):
        self._timer.cancel()
        ok = 0
        print('\n──────────────────────────────────')
        print('  AEROKNOW L01 — STACK VERIFICATION')
        print('──────────────────────────────────')
        for topic, _, desc in REQUIRED_TOPICS:
            status = '✓ OK ' if self.received[topic] else '✗ FAIL'
            if self.received[topic]:
                ok += 1
            print(f'  {status}  {topic}')
            if not self.received[topic]:
                print(f'         → Check: is MAVROS2 running? (ros2 node list | grep mavros)')
        print('──────────────────────────────────')
        print(f'  Result: {ok}/{len(REQUIRED_TOPICS)} topics alive')
        if ok == len(REQUIRED_TOPICS):
            print('  STATUS: STACK OK — ready for L02\n')
        else:
            print('  STATUS: PROBLEMS FOUND — see errors above\n')
        rclpy.shutdown()


def main():
    rclpy.init()
    node = SanityCheck()
    rclpy.spin(node)

if __name__ == '__main__':
    main()

Common Errors & Fixes

These are the errors that reliably appear during L01 setup — documented here so you spend minutes, not hours, recovering from them.

Error 01 — Gazebo fails to open / black window ERROR [gz]: Unable to initialize rendering engine. Aborting.

Cause: Gazebo Garden requires OpenGL 3.3+. Inside VirtualBox, 3D Acceleration must be enabled in VM Settings → Display. On bare metal with NVIDIA, make sure the proprietary driver is active (nvidia-smi should print GPU info). Fix: sudo ubuntu-drivers autoinstall then reboot. If on VirtualBox, enable "Enable 3D Acceleration" and set video memory ≥ 128 MB.

Error 02 — MAVROS2 connection timeout FCU: timeout, no heartbeat received in 3 seconds

Cause: MAVROS2 cannot reach PX4 on UDP port 14550. Either PX4 SITL is not running, or a firewall is blocking local UDP. Fix: Confirm PX4 SITL is running and shows Ready for takeoff! in Terminal 1. Then check: sudo ufw status — if UFW is active, run sudo ufw allow from 127.0.0.1.

Error 03 — colcon build fails with "mavros_msgs not found" CMake Error: Could not find package configuration file for "mavros_msgs"

Cause: MAVROS2 is installed but not sourced into the current shell. Fix: Run source /opt/ros/humble/setup.bash before colcon build, or add it to ~/.bashrc if you haven't already. Then re-run colcon build.

Error 04 — PX4 make fails: "No such file or directory: px4_sitl_default" make: *** [Makefile:232: px4_sitl_default] Error 1

Cause: The PX4 git submodules were not initialised. Fix: cd ~/PX4-Autopilot && git submodule update --init --recursive. This downloads ~500 MB of additional source — use the --recursive flag or nested submodules will also be missing.

Error 05 — sanity_check shows GPS fail despite everything running ��� FAIL /mavros/global_position/global

Cause: The simulated GPS takes 10–20 seconds to acquire a 3D fix after PX4 SITL starts. This is intentional — it mirrors real GPS cold-start behaviour. Fix: Wait 20 seconds after MAVROS2 connects, then re-run sanity_check. You can watch GPS status with: ros2 topic echo /mavros/state — wait until system_status: 4 (MAV_STATE_ACTIVE).

Extension Challenges

You have a working stack. Here are three harder things to try before moving to L02.

  1. Extend sanity_check with latency measurement. Record the timestamp of the first message received on each topic and compute the gap between MAVROS2 startup and first valid IMU reading. Is it consistent across restarts? Hint: use self.get_clock().now() and store a startup time in __init__. This skill — measuring topic latency in ROS2 — is used in L05 when we tune the EKF timing.
  2. Watch the rqt_graph. With PX4, Gazebo, MAVROS2, and sanity_check all running simultaneously, open a fourth terminal and run rqt_graph. You should see the MAVROS2 node connected to your sanity_check node via topic edges. This is the exact same graph visualisation used in L02 — ROS2 Fundamentals to reason about node architecture.
  3. Command a manual land from the PX4 shell. After the drone is hovering from Step 8, type commander land in the PX4 shell prompt. Observe the landing sequence. Then look at ros2 topic echo /mavros/state and watch the mode field change as PX4 transitions from POSCTL to LAND. Understanding mode transitions is the first concept in L03 — PX4 + MAVROS2.
