The SDF Format — Gazebo's World Description Language

Every Gazebo world is described in an SDF (Simulation Description Format) file — XML that specifies every object, its physics properties, and its sensor plugins. Before writing any SDF, you need to understand its six key building blocks.

<world>

The root element. Contains global settings: gravity, magnetic field, atmosphere, and the plugin list. Everything else lives inside it. One SDF file = one world.

<model>

A rigid body entity — a building, a vehicle, the drone. Has a name, a pose, and one or more <link> elements. Models can be static (fixed) or dynamic (physics-simulated).

<link>

The physical part of a model. Defines mass, inertia, collision geometry, and visual geometry. A drone has many links — body, rotors, camera mount.

<sensor>

A measurement device attached to a link. Types: camera, depth_camera, gpu_lidar, imu. Each sensor has its own plugin that publishes data to a topic.

<plugin>

C++ code that runs inside the simulation. World-level plugins add behaviour (wind, physics, time). Sensor plugins convert simulation data to topics. Model plugins add controller logic.

<pose>

Position and orientation: x y z roll pitch yaw in metres and radians. All poses are relative to the parent element unless relative_to is specified. Yaw=0 points along the X axis (East in Gazebo).

Gazebo vs. ROS2 Coordinate Frames

This difference causes confusion every time. It is worth knowing before you write a single SDF line.

Frame Convention Mismatch Gazebo uses ENU — X East, Y North, Z Up. PX4 / MAVLink uses NED — X North, Y East, Z Down. MAVROS2 automatically converts between them on all topics. Your SDF poses are in Gazebo ENU. Your ROS2 setpoints are in ENU (MAVROS2 local frame). The PX4 shell displays NED internally. When you set a building at Gazebo pose (10 0 0) it is 10 m East, which is Y=10 in PX4's NED frame. The sensor_checker.py in Step 08 displays ENU so it matches what you see in Gazebo.

Step 03–04: Sensor Plugins — What They Are and How to Configure Them

A sensor plugin is a Gazebo module that reads simulated physics data and publishes it as a Gazebo topic. The ros_gz_bridge then translates those Gazebo topics into ROS2 topics your nodes can subscribe to. The chain is: physics engine → sensor plugin → Gazebo topic → bridge → ROS2 topic.

Step 04: Understanding the Wind Plugin

The wind plugin is already in the world SDF from Step 02. This section explains what it does and how to tune it for different test scenarios.

STEP 04 Tune wind parameters for your scenario
What the wind plugin parameters mean

The Gazebo WindEffects plugin applies a stochastic horizontal force to all dynamic bodies in the world. It models a sustained wind component with random gust perturbations layered on top. The parameters in the world file set the gust characteristics; the sustained wind direction is set separately.

To add a sustained 5 m/s northward wind (in addition to gusts), add this inside the <wind> element of the world file. Open aeroknow.world and add this block immediately after the closing </physics> tag:

xml · add to aeroknow.world (after </physics>) 
<!-- Sustained wind: 5 m/s northward (y=5 in ENU = North) -->
<wind>
  <linear_velocity>0 5 0</linear_velocity>  <!-- ENU: x=East y=North z=Up -->
</wind>

Step 07: world_spawner.py — Dynamic Object Spawning

The world file defines static buildings. But for L06 and L08 testing, you need to spawn and despawn target objects at runtime — without restarting the simulation. The world spawner uses the Gazebo service API via ROS2 to inject new models.

STEP 07 Write world_spawner.py
Why runtime spawning instead of editing the world file

In test missions you want to place targets at specific positions, observe whether the detection pipeline finds them, then move them — without restarting the 2-minute simulation boot sequence each time. The Gazebo /world/aeroknow/create service accepts an SDF string and spawns any model described in it. The /world/aeroknow/remove service despawns by name. Both are accessible from ROS2 via the ros_gz_interfaces package.

bash 
$ sudo apt install -y ros-humble-ros-gz-interfaces
$ nano ~/aeroknow_ws/src/aeroknow_sim/aeroknow_sim/world_spawner.py

"""
world_spawner.py
────────────────
Spawns and despawns objects in the running Gazebo world
without restarting the simulation.

Subscriptions:
  /world_spawner/spawn_barrel  (std_msgs/String) — "name x y z"
  /world_spawner/despawn       (std_msgs/String) — model name

Used in L06 to place detection targets at arbitrary positions.
"""

from rclpy.node import Node
from rclpy.qos  import QoSProfile, ReliabilityPolicy, HistoryPolicy

# SDF template for a red barrel. {name}, {x}, {y}, {z} are formatted in at spawn time.

  <model name='{name}'>
    <static>true</static>
    <pose>{x} {y} {z} 0 0 0</pose>

      <collision name='col'>

      </collision>

    </link>
  </model>
</sdf>"""


class WorldSpawner(Node):

    def __init__(self):
        super().__init__('world_spawner')

        rel = QoSProfile(reliability=ReliabilityPolicy.RELIABLE,
                         history=HistoryPolicy.KEEP_LAST, depth=10)

        # Topic to spawn a barrel: publish "my_barrel 15.0 20.0 0.5"
        self.create_subscription(String, '/world_spawner/spawn_barrel',
                                 self.spawn_cb, rel)

        # Topic to despawn any model by name
        self.create_subscription(String, '/world_spawner/despawn',
                                 self.despawn_cb, rel)

        # Gazebo world create / delete service clients
        # These are provided by ros_gz_bridge when it is running.
        self.spawn_cli   = self.create_client(SpawnEntity,  '/world/aeroknow/create')
        self.despawn_cli = self.create_client(DeleteEntity, '/world/aeroknow/remove')

        self.spawned_models = set()
        self.get_logger().info(
            'WorldSpawner ready.\n'
            '  Spawn: ros2 topic pub --once /world_spawner/spawn_barrel std_msgs/msg/String "data: \'barrel_1 20 15 0.5\'"\n'
            '  Remove: ros2 topic pub --once /world_spawner/despawn std_msgs/msg/String "data: \'barrel_1\'"'
        )

    def spawn_cb(self, msg: String):
        """
        Parse "name x y z" and spawn a red barrel at that position.
        """
        parts = msg.data.strip().split()
        if len(parts) != 4:
            self.get_logger().warn(f'spawn_barrel: expected "name x y z", got: {msg.data}')
            return

        name, x, y, z = parts[0], float(parts[1]), float(parts[2]), float(parts[3])

        if not self.spawn_cli.wait_for_service(timeout_sec=3.0):
            self.get_logger().error('Gazebo spawn service not available — is ros_gz_bridge running?')
            return

        req = SpawnEntity.Request()
        req.xml  = BARREL_SDF_TEMPLATE.format(name=name, x=x, y=y, z=z)
        req.name = name
        self.spawn_cli.call_async(req)
        self.spawned_models.add(name)
        self.get_logger().info(f'Spawned barrel "{name}" at ({x}, {y}, {z})')

    def despawn_cb(self, msg: String):
        """Remove a model by name."""
        name = msg.data.strip()
        if not self.despawn_cli.wait_for_service(timeout_sec=3.0):
            self.get_logger().error('Gazebo delete service not available')
            return
        req = DeleteEntity.Request()
        req.name = name
        self.despawn_cli.call_async(req)
        self.spawned_models.discard(name)
        self.get_logger().info(f'Despawned "{name}"')


def main(args=None):
    rclpy.init(args=args)
    node = WorldSpawner()
    try: rclpy.spin(node)
    except KeyboardInterrupt: pass
    finally: node.destroy_node(); rclpy.shutdown()

if __name__ == '__main__': main()

Step 08: sensor_checker.py — Verify All Sensor Topics

Before L05 and L06 depend on sensor data, you need a reliable way to confirm every sensor is publishing at the expected rate with valid data. The sensor checker is the L04 equivalent of L01's sanity_check — it verifies the full new sensor stack in a single command.

STEP 08 Write sensor_checker.py
bash 
$ nano ~/aeroknow_ws/src/aeroknow_sim/aeroknow_sim/sensor_checker.py
python · sensor_checker.py 
#!/usr/bin/env python3
"""
sensor_checker.py
─────────────────
Verifies all L04 sensor topics are alive and publishing at expected rates.
Runs for 10 seconds, measures each topic's actual publish rate, then
reports PASS / WARN / FAIL per sensor with a summary.
"""

import rclpy
from rclpy.node import Node
from rclpy.qos  import QoSProfile, ReliabilityPolicy, HistoryPolicy

from sensor_msgs.msg import Image, LaserScan, Imu, NavSatFix, BatteryState
from mavros_msgs.msg import State

import time


# Sensor definitions: (topic, message_type, min_hz, qos_profile_key)
# min_hz:  minimum acceptable publish rate
# qos:     'sensor' = BEST_EFFORT, 'reliable' = RELIABLE
SENSORS = [
    ('/camera/image_raw',              Image,       20,  'sensor'),
    ('/depth_camera/depth',             Image,       10,  'sensor'),
    ('/scan',                            LaserScan,   8,   'sensor'),
    ('/mavros/imu/data',                 Imu,         50,  'sensor'),
    ('/mavros/global_position/global',   NavSatFix,   3,   'sensor'),
    ('/mavros/battery',                  BatteryState,0.5, 'sensor'),
    ('/mavros/state',                    State,       0.5, 'reliable'),
]

CHECK_DURATION_S = 10.0


class SensorChecker(Node):

    def __init__(self):
        super().__init__('sensor_checker')

        sen_qos = QoSProfile(reliability=ReliabilityPolicy.BEST_EFFORT,
                             history=HistoryPolicy.KEEP_LAST, depth=1)
        rel_qos = QoSProfile(reliability=ReliabilityPolicy.RELIABLE,
                             history=HistoryPolicy.KEEP_LAST, depth=10)

        # Count messages per topic
        self.counts = {topic: 0 for topic, _, _, _ in SENSORS}
        self.t_start = time.time()

        for topic, msg_type, _, qos_key in SENSORS:
            qos = sen_qos if qos_key == 'sensor' else rel_qos
            self.create_subscription(
                msg_type, topic,
                lambda msg, t=topic: self._count(t),
                qos
            )

        # Report after CHECK_DURATION_S seconds
        self.create_timer(CHECK_DURATION_S, self.report)
        print(f'\nChecking sensors for {CHECK_DURATION_S:.0f}s...\n')

    def _count(self, topic: str):
        self.counts[topic] += 1

    def report(self):
        elapsed = time.time() - self.t_start

        print('─'*62)
        print(f'  AEROKNOW L04 — SENSOR STACK VERIFICATION')
        print('─'*62)
        print(f'  {"Topic":<38} {"Rate":>6} {"Min":>5} {"Status"}')
        print('─'*62)

        all_ok = True
        for topic, _, min_hz, _ in SENSORS:
            count = self.counts[topic]
            hz    = count / elapsed
            if hz >= min_hz:
                status = '\033[92m✓ OK  \033[0m'
            elif hz >= min_hz * 0.5:
                status = '\033[93m⚠ SLOW\033[0m'
                all_ok = False
            else:
                status = '\033[91m✗ FAIL\033[0m'
                all_ok = False
            print(f'  {topic:<38} {hz:>5.1f}Hz {min_hz:>4}Hz {status}')

        print('─'*62)
        overall = '\033[92mSENSOR STACK OK — ready for L05\033[0m' if all_ok \
                  else '\033[91mSOME SENSORS FAILED — check ros_gz_bridge is running\033[0m'
        print(f'  {overall}')
        print('─'*62 + '\n')

        rclpy.shutdown()


def main(args=None):
    rclpy.init(args=args)
    node = SensorChecker()
    try: rclpy.spin(node)
    except (KeyboardInterrupt, Exception): pass
    finally:
        if rclpy.ok(): node.destroy_node(); rclpy.shutdown()


if __name__ == '__main__': main()

Step 09: The L04 Launch File — Wiring Everything Together

The launch file does three things: starts MAVROS2, starts the ros_gz_bridge (translating Gazebo sensor topics to ROS2), and starts the world_spawner and sensor_checker nodes. The bridge is the critical new piece — without it, all the sensor SDF work above produces Gazebo-internal topics that your ROS2 nodes cannot see.

STEP 09 Write l04_launch.py
How the bridge topic mapping works

The ros_gz_bridge node takes a list of topic@GzType@RosType strings. For each pair, it subscribes to the Gazebo topic and republishes as the equivalent ROS2 message type. The @ separator means "bridge in both directions" — [ means Gazebo → ROS2 only, ] means ROS2 → Gazebo only. All sensor bridges are one-directional Gazebo → ROS2, so we use [.

bash 
$ nano ~/aeroknow_ws/src/aeroknow_sim/launch/l04_launch.py
python · l04_launch.py 
#!/usr/bin/env python3
"""
l04_launch.py
─────────────
Starts: MAVROS2 + ros_gz_bridge (sensor topics) + world_spawner + sensor_checker.

Assumes PX4 SITL is already running (Terminal 1: sitl_aeroknow).

Usage:
  ros2 launch aeroknow_sim l04_launch.py
"""

from launch import LaunchDescription
from launch.actions import LogInfo, TimerAction
from launch_ros.actions import Node


def generate_launch_description():

    # ── MAVROS2 ────────────────────────────────────────────────────
    # Same as before — connects to PX4 SITL over UDP
    mavros_node = Node(
        package='mavros',
        executable='mavros_node',
        name='mavros',
        output='screen',
        parameters=[{
            'fcu_url':        'udp://:14540@127.0.0.1:14557',
            'gcs_url':        '',
            'target_system_id':    1,
            'target_component_id': 1,
        }],
    )

    # ── Gazebo → ROS2 Bridge ────────────────────────────────────────
    # Each entry: "gz_topic@gz_type@ros_type"
    # '[' means Gazebo → ROS2 (unidirectional)
    # '@' means bidirectional
    #
    # Camera image: Gazebo publishes gz.msgs.Image, bridge converts to sensor_msgs/Image.
    # CameraInfo:   Needed for L06 camera calibration matrix.
    # DepthImage:   Gazebo publishes gz.msgs.Image with float depth.
    # LaserScan:    Gazebo publishes gz.msgs.LaserScan → sensor_msgs/LaserScan.
    # WindSpeed:    World wind vector (for monitoring / L05 disturbance estimation).
    bridge_node = Node(
        package='ros_gz_bridge',
        executable='parameter_bridge',
        name='gz_ros_bridge',
        output='screen',
        arguments=[
            # Downward camera image
            '/camera/image[gz.msgs.Image@sensor_msgs/msg/Image',
            # Camera calibration info (needed for L06 projection)
            '/camera/camera_info[gz.msgs.CameraInfo@sensor_msgs/msg/CameraInfo',
            # Depth image
            '/depth_camera/depth[gz.msgs.Image@sensor_msgs/msg/Image',
            # 2D lidar scan
            '/scan[gz.msgs.LaserScan@sensor_msgs/msg/LaserScan',
            # Gazebo clock → ROS2 clock (keeps timestamps in sync)
            '/clock[gz.msgs.Clock@rosgraph_msgs/msg/Clock',
        ],
        remappings=[
            # Remap Gazebo topic names to cleaner ROS2 names
            ('/camera/image',      '/camera/image_raw'),
            ('/depth_camera/depth', '/depth_camera/depth'),
        ],
    )

    # ── World Spawner — starts 2s after bridge (bridge must be up first) ──
    spawner_node = TimerAction(
        period=2.0,
        actions=[Node(
            package='aeroknow_sim',
            executable='world_spawner',
            name='world_spawner',
            output='screen',
        )]
    )

    # ── L02 Bringup (telemetry, health, mission action) ────────────
    bringup_nodes = [
        Node(package='aeroknow_bringup', executable='telemetry_listener',   name='telemetry_listener',   output='screen'),
        Node(package='aeroknow_bringup', executable='heartbeat_service',    name='heartbeat_service',    output='log'),
        Node(package='aeroknow_bringup', executable='mission_action_server', name='mission_action_server', output='screen'),
        Node(package='aeroknow_bringup', executable='mode_manager',         name='mode_manager',         output='log'),
        Node(package='aeroknow_bringup', executable='emergency_stop',       name='emergency_stop',       output='screen'),
    ]

    return LaunchDescription([
        LogInfo(msg='[AEROKNOW L04] Starting sensor world stack...'),
        mavros_node,
        bridge_node,
        spawner_node,
        *bringup_nodes,
    ])

Common Errors & Fixes

Error 01 — Camera topic not visible after bridge starts ros2 topic list | grep camera → (nothing)

Cause: The Gazebo sensor plugin did not start because the gz-sim-sensors-system plugin is missing from the world SDF. Without it, no sensor plugins fire regardless of what the model SDF says. Fix: Confirm the world SDF has this block inside <world>: <plugin filename="gz-sim-sensors-system" name="gz::sim::systems::Sensors"><render_engine>ogre2</render_engine></plugin>. The ogre2 render engine is required for camera sensors — without it, depth and RGB cameras will not render.

Error 02 — x500_sensors model not found by PX4 ERROR [gz]: Model 'x500_sensors' not found in resource path

Cause: GZ_SIM_RESOURCE_PATH does not include the models directory, or the symlink in ~/.gz/models/ is broken. Fix: (1) Run echo $GZ_SIM_RESOURCE_PATH — it should contain ~/aeroknow_ws/src/aeroknow_sim/models. If not, run source ~/.bashrc. (2) Check the symlink: ls -la ~/.gz/models/x500_sensors — it should point to the models directory. If it shows "broken link", re-run the ln -sf command from Step 05.

Error 03 — Bridge starts but camera images are black/empty camera/image_raw publishes at 30Hz but image data is all zeros

Cause: The Gazebo render engine did not initialise correctly — usually happens on headless servers or VMs with software rendering. Fix: Add the LIBGL_ALWAYS_SOFTWARE=1 environment variable before the PX4 launch: LIBGL_ALWAYS_SOFTWARE=1 sitl_aeroknow. This forces Mesa software rendering — images will be valid but Gazebo performance will drop 30–50%. On a physical machine with a GPU, confirm the driver is loaded: glxinfo | grep renderer should show your GPU name, not "llvmpipe".

Error 04 — world_spawner service not available Gazebo spawn service not available — is ros_gz_bridge running?

Cause: The spawn and delete services exposed by Gazebo Garden use a different interface package than ROS2 services. The ros_gz_interfaces package must be installed, and the world name in the service path must match exactly. Fix: (1) Confirm: sudo apt install ros-humble-ros-gz-interfaces. (2) Confirm your world is named aeroknow in the SDF (<world name="aeroknow">) — the service path /world/aeroknow/create uses this exact name. (3) List available services: ros2 service list | grep world to see what Gazebo is actually exposing.

Error 05 — Drone model is missing rotors / behaves incorrectly model x500_sensors spawned but drone falls immediately or has no thrust

Cause: The <include> merge of the base x500 model failed — either the URI model://x500 is not found by Gazebo, or PX4's model search path is not set. Fix: Add the PX4 models directory to the resource path: export GZ_SIM_RESOURCE_PATH=$GZ_SIM_RESOURCE_PATH:~/PX4-Autopilot/Tools/simulation/gz/models (and add to ~/.bashrc). This tells Gazebo where to find the base x500 model that our x500_sensors inherits from.

Extension Challenges

The sensor world is running. Here are three harder things to try before L05.

  1. Add a radar altimeter plugin to the drone model. A radar altimeter measures height above the ground below — unlike the barometer which measures absolute pressure. In Gazebo, this is an altimeter sensor type. Add it to model.sdf, bridge it to the ROS2 topic /mavros/altitude, and extend sensor_checker.py to verify it. The radar altimeter topic will be used in L05 as a GNSS-denied height reference when the barometer drifts.
  2. Add a second building using world_spawner at runtime. While the L04 stack is running, use the spawn topic to place a new tall building (make it a box 8×8×30 m) at position (60, 0, 15). Watch it appear in Gazebo without restarting the simulation. Then verify the lidar scan topic shows a new obstacle bearing in the expected direction. Hint: you'll need to send a custom SDF string using ros2 topic pub --once /world_spawner/spawn_barrel won't work for a box — extend world_spawner.py with a spawn_box topic that accepts "name x y z sx sy sz".
  3. Visualise camera and lidar data in RViz2. Open RViz2 (ros2 run rviz2 rviz2), add an Image display subscribed to /camera/image_raw, and a LaserScan display subscribed to /scan. You should see the camera feed from the drone's perspective and a 360° circle of range measurements from the lidar. Save the RViz configuration to ~/aeroknow_ws/src/aeroknow_sim/config/l04_sensors.rviz and add it to the launch file so it opens automatically. This is the same visualisation used in L05 to monitor the GNSS-denied navigation state.
