Deep Learning & UAV Perception

A UAV that cannot interpret what its sensors see is blind. Deep learning gives UAVs the ability to extract meaning from raw images, point clouds, and video streams — identifying targets, estimating depth, tracking motion, and understanding the 3D structure of the environment. This module covers CNNs, object detection, Vision Transformers, optical flow, and depth estimation — with a sharp focus on what works in the constrained compute and harsh conditions of military platforms.

7.1 The UAV Perception Problem

Plain English First

A camera mounted on a UAV produces a stream of numbers — pixel brightness values. Nothing in those numbers directly says "this is a vehicle" or "that is a person." Turning raw pixel data into useful information — locations of targets, distances to obstacles, velocity of moving objects — is the perception problem. It is the bridge between sensing the world and acting in it. Before deep learning, this required hand-crafting features (edge detectors, histogram gradients) that were brittle, slow, and needed re-engineering for every new environment. Deep learning learns those features automatically from data, and the resulting systems are both more accurate and more adaptable.

The perception stack for a military UAV involves several distinct tasks, each addressed by a different deep learning architecture: MDPI Autonomous Control Ch.5

The UAV-Specific Perception Challenges Standard deep learning benchmarks (ImageNet, COCO) use ground-level imagery. UAV imagery is fundamentally different: extreme viewing angles (near-nadir), small object size (a person is 10–20 pixels from 100 m altitude), high motion blur, dynamic lighting, and dense clutter. Models trained on ground-level data frequently fail when deployed on UAVs without fine-tuning on aerial data. Bestaoui Sebbane §9.5

7.2 Convolutional Neural Networks — The Spatial Feature Extractor

Plain English First

A Convolutional Neural Network (CNN) processes images by sliding small filters (kernels) across the image and detecting local patterns — edges, corners, textures, then shapes, then objects at increasing abstraction. Think of it like looking at an image through increasingly complex magnifying glasses: the first glass sees edges, the second sees shapes made of edges, the third sees objects made of shapes. This hierarchical feature extraction is what makes CNNs so powerful for vision — and it is the reason every modern UAV perception system is built on a CNN backbone or its attention-based successor, the Vision Transformer.

The Convolution Operation

A 2D convolution slides a kernel $\mathbf{k}$ across an input feature map $\mathbf{x}$ to produce an output feature map: MDPI Autonomous Control Ch.5

2D Discrete Convolution (Feature Map) $$(\mathbf{x} * \mathbf{k})[i,j] = \sum_{m=0}^{K-1}\sum_{n=0}^{K-1} \mathbf{x}[i+m,\, j+n]\cdot\mathbf{k}[m,n]$$

$K$: kernel size (typically 3 or 5); $[i,j]$: output pixel location. For an input of size $H \times W$ with a $K \times K$ kernel, stride $s$, and padding $p$, the output size is:

$$H_{out} = \left\lfloor\frac{H - K + 2p}{s}\right\rfloor + 1, \qquad W_{out} = \left\lfloor\frac{W - K + 2p}{s}\right\rfloor + 1$$

With padding $p = (K-1)/2$ ("same" padding), output size equals input size. With stride $s = 2$ (strided convolution), output halves in size — used for downsampling instead of max pooling in modern architectures. Number of learnable parameters per conv layer: $(K^2 \cdot C_{in} + 1) \times C_{out}$, where $C_{in}, C_{out}$ are input/output channel counts and $+1$ is the bias term.

Key CNN Architectural Innovations

Residual Connections (ResNet)
$\mathbf{y} = \mathcal{F}(\mathbf{x}) + \mathbf{x}$
Identity shortcut bypasses layers. Allows gradients to flow directly, enabling networks 100+ layers deep. ResNet-50 backbone is the most common starting point for UAV perception.
Depthwise Separable Conv
Factorises $K^2 C_{in} C_{out}$ operations into $K^2 C_{in} + C_{in} C_{out}$. 8–9× fewer operations. Used in MobileNet — the backbone of choice for UAV edge deployment (Jetson Orin, 15W budget).
Feature Pyramid Network
Multi-scale feature maps fused top-down. Detects small objects (20–40px, which is a person at 100 m UAV altitude) using high-resolution shallow features, while using deep features for semantic context. Essential for aerial detection.
Batch Normalisation
Normalises layer inputs to zero mean, unit variance per batch. $\hat{x} = (x-\mu_B)/\sigma_B$, then $y = \gamma\hat{x} + \beta$. Dramatically stabilises training, allows 10× higher learning rates, provides mild regularisation. Now standard in every backbone.

7.3 Object Detection — Finding and Classifying Targets

Plain English First

Object detection solves two problems at once: where is the object in the image (a bounding box), and what is it (a class label). For a military UAV, this might mean detecting vehicles, personnel, weapons systems, or infrastructure in real-time aerial imagery. The challenge is that objects of interest can be tiny (10–20 pixels from altitude), partially occluded, camouflaged, or appear at any orientation — none of which standard ground-level detectors handle well.

Detection Paradigms: One-Stage vs Two-Stage

ParadigmMechanismSpeedAccuracyUAV Use
Two-stage (R-CNN family) Stage 1: propose regions. Stage 2: classify + refine each region ~5–15 FPSHigh mAP Offline ISR analysis; not real-time
One-stage (YOLO family) Single network predicts all boxes and classes simultaneously 30–150+ FPSCompetitive mAP Real-time onboard detection
Transformer (DETR family) Attention-based end-to-end detection; no NMS post-processing ~15–30 FPSState of the art High-accuracy ISR on capable hardware

YOLO — You Only Look Once

YOLO frames detection as a regression problem: a single CNN produces a grid of predictions directly. MDPI Autonomous Control Ch.5

YOLOv8 Output Tensor — Grid Cell Prediction $$\text{Output} \in \mathbb{R}^{G \times G \times (5 + C)}$$ $$\text{per cell: } [t_x, t_y, t_w, t_h, p_{obj}, p_1, p_2, \ldots, p_C]$$ $$b_x = \sigma(t_x) + c_x, \quad b_y = \sigma(t_y) + c_y, \quad b_w = p_w\,e^{t_w}, \quad b_h = p_h\,e^{t_h}$$

$G \times G$: grid size (e.g., 20×20 at the smallest feature map scale); $c_x, c_y$: grid cell offset; $p_w, p_h$: anchor box prior dimensions. $\sigma(t_x), \sigma(t_y)$: sigmoid constrains box centre within cell. $p_{obj}$: objectness score — is there an object here at all? $p_c$: class probabilities. Final confidence = $p_{obj} \times p_c$. Multi-scale prediction (3 grid scales in YOLOv8) handles objects at different sizes.

YOLO Loss Function

YOLOv8 Loss — Combined Localisation + Classification + Objectness $$\mathcal{L} = \lambda_{box}\mathcal{L}_{IoU} + \lambda_{cls}\mathcal{L}_{BCE,cls} + \lambda_{dfl}\mathcal{L}_{DFL}$$ $$\mathcal{L}_{IoU} = 1 - \text{IoU}(b_{pred}, b_{gt}) \quad \text{where IoU} = \frac{|b_{pred} \cap b_{gt}|}{|b_{pred} \cup b_{gt}|}$$

$\mathcal{L}_{IoU}$: Intersection-over-Union loss — directly optimises box overlap quality. $\mathcal{L}_{BCE,cls}$: binary cross-entropy for each class independently (allows multi-label). $\mathcal{L}_{DFL}$: Distribution Focal Loss — predicts a distribution over box edge locations rather than a single value, improving localisation precision for small targets. Default weights: $\lambda_{box} = 7.5$, $\lambda_{cls} = 0.5$, $\lambda_{dfl} = 1.5$. Bestaoui Sebbane §9.6

Detection Metrics — mAP, Precision, Recall

Mean Average Precision (mAP) — The Standard Benchmark Metric $$\text{IoU} = \frac{|b_{pred} \cap b_{gt}|}{|b_{pred} \cup b_{gt}|} \qquad \text{(detection match criterion: IoU} \geq 0.5\text{)}$$ $$\text{AP}_c = \int_0^1 p(r)\,dr \approx \sum_{k} p(k)\,\Delta r(k) \qquad \text{(area under precision-recall curve)}$$ $$\text{mAP} = \frac{1}{C}\sum_{c=1}^C \text{AP}_c$$

A detection is a True Positive if IoU with a ground-truth box exceeds 0.5 (COCO uses 0.5–0.95). mAP@0.5:0.95 is the standard COCO benchmark — average AP over 10 IoU thresholds. For military UAV applications, a high-recall / moderate-precision operating point is often preferred (miss nothing, accept some false alarms) vs. the reverse — this is controlled by the confidence threshold.

Worked Example 7.2 — Small Object Detection at UAV Altitude

An MQ-9 carries an EO sensor at 7,620 m altitude with GSD = 24 cm/pixel (from Module 01). A standard military vehicle (Toyota Hilux technical) is 5.3 m long × 1.8 m wide. How many pixels does it occupy? What does this imply for detector design?

Vehicle dimensions in pixels: Length: $5.3 / 0.24 \approx 22$ px. Width: $1.8 / 0.24 \approx 8$ px. Bounding box area: $22 \times 8 = 176$ px².

COCO "small object" threshold: area $< 32^2 = 1024$ px². At 176 px², this vehicle falls into the "small" category — the hardest regime for all detectors. YOLOv8 achieves mAP@0.5 of ~54% on COCO small objects vs ~82% on large objects.

Mitigations for small-object detection: (1) Higher input resolution (1280×1280 instead of 640×640) — doubles compute but improves small AP by ~8%. (2) SAHI (Sliced Aided Hyper Inference) — run detector on overlapping tiles, merge results. (3) Feature Pyramid Network with higher-resolution P2 feature map. (4) Fine-tune on aerial vehicle dataset (VisDrone, UAVDT) — domain adaptation from COCO. At lower altitudes (2,000 m), GSD ≈ 6 cm/pixel, vehicle = 88×30 px — "medium" object, standard YOLO achieves ~72% mAP. Mission altitude is a key perception design parameter. Bestaoui Sebbane §9.7

7.4 Vision Transformers (ViT) — Attention Over Patches

Plain English First

CNNs process images locally — each filter sees a small patch at a time. For many tasks this works well, but sometimes context far away in the image is important: a camouflaged vehicle is more recognisable when you also see the nearby road and tyre tracks. Vision Transformers address this by splitting the image into patches and letting every patch "attend" to every other patch — enabling global context from the very first layer. They have surpassed CNNs on many benchmarks but are computationally heavier, so hardware context matters when deciding which to use.

The Self-Attention Mechanism

Self-attention is the core operation in a Transformer. For an image split into $N$ patches of size $P \times P$: MDPI Autonomous Control Ch.6

Scaled Dot-Product Self-Attention $$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right) V$$ $$Q = X W_Q, \quad K = X W_K, \quad V = X W_V$$

$X \in \mathbb{R}^{N \times d}$: sequence of $N$ patch embeddings of dimension $d$. $W_Q, W_K, W_V \in \mathbb{R}^{d \times d_k}$: learned projection matrices. $Q$: queries ("what am I looking for?"); $K$: keys ("what do I contain?"); $V$: values ("what do I send?"). The softmax score $\frac{QK^T}{\sqrt{d_k}}$ for patch $i$ vs patch $j$ is how much patch $i$ should "pay attention" to the content of patch $j$. Dividing by $\sqrt{d_k}$ prevents vanishing gradients in the softmax from very large dot products.

Theorem 7.1 — Attention Complexity and Its UAV Implications

Standard self-attention has quadratic complexity in the sequence length:

$$\text{Complexity of full self-attention:} \quad O(N^2 d)$$ $$\text{For a 640×640 image with P=16 patches: } N = (640/16)^2 = 1600 \text{ patches}$$ $$\Rightarrow 1600^2 = 2.56 \times 10^6 \text{ attention weight pairs per head per layer}$$

With 12 attention heads and 12 layers, this is $\sim 360M$ operations per forward pass — approximately $8\times$ more expensive than a ResNet-50. Practical consequence for UAVs: Full ViT (ViT-B/16) runs at ~5–8 FPS on a Jetson Orin Nano — unsuitable for real-time detection. Solutions: (1) Hierarchical ViTs (Swin Transformer) — attention only within local windows, $O(N \cdot w^2 d)$ where $w$ is window size; (2) Hybrid architectures — CNN backbone + Transformer neck; (3) Distillation — compress ViT into a smaller CNN for deployment. Bestaoui Sebbane §10.1

Swin Transformer — Making ViT Practical

The Swin Transformer (Shifted Window Transformer) reduces attention complexity from $O(N^2)$ to $O(N)$ by restricting attention to local windows, then shifting the windows between layers to allow cross-window communication:

Swin Window Attention Complexity $$\text{W-MSA complexity:} \quad O\!\left(4N d^2 + 2Nw^2 d\right) \approx O(Nw^2 d)$$

$w$: window size (typically 7×7 patches = 49 tokens per window). For $N = 1600$ patches, $w=7$: complexity ≈ $1600 \times 49 \times d = 78400d$ vs. $2.56 \times 10^6 d$ for full attention — a 33× reduction. Swin-T achieves 81.3% ImageNet top-1 at ~4.5 GFLOPs (comparable to ResNet-50), and runs at ~35 FPS on Jetson Orin Nano — borderline real-time for UAV use.

7.5 Optical Flow — Seeing Motion

Plain English First

Optical flow is the apparent motion of pixels between two consecutive video frames. It tells you: every pixel moved this many pixels left/right/up/down between frame $t$ and frame $t+1$. For a UAV flying forward, the entire image flows backward — that flow pattern tells you speed and direction. For a vehicle on the ground moving while the UAV is stationary, only the vehicle's pixels flow — that's how you detect moving targets. Optical flow is a bridge between raw video and motion understanding, and is fundamental to both visual ego-motion estimation (VIO, Module 04) and moving target detection.

The Lucas-Kanade Equation — Classical Foundation

Classical optical flow is derived from the brightness constancy assumption: Bestaoui Sebbane §10.2

Optical Flow Constraint Equation $$I_x u + I_y v + I_t = 0$$ $$\begin{bmatrix}I_x & I_y\end{bmatrix}\begin{bmatrix}u\\v\end{bmatrix} = -I_t$$

$I_x, I_y$: spatial image gradients (edges); $I_t$: temporal gradient (change over time); $u, v$: unknown flow at pixel $(x,y)$. This is one equation with two unknowns — the aperture problem: without additional constraints, only motion perpendicular to edges is observable. Lucas-Kanade solves this by assuming constant flow within a local window $\Omega$: $$\min_{u,v} \sum_{(x,y)\in\Omega} (I_x u + I_y v + I_t)^2 \quad \Rightarrow \quad \underbrace{\begin{bmatrix}\sum I_x^2 & \sum I_x I_y \\ \sum I_x I_y & \sum I_y^2\end{bmatrix}}_{M} \begin{bmatrix}u\\v\end{bmatrix} = -\begin{bmatrix}\sum I_x I_t \\ \sum I_y I_t\end{bmatrix}$$ The matrix $M$ is the structure tensor. Its eigenvalues determine whether optical flow is reliably estimable at this point.

Deep Optical Flow — RAFT

Deep learning methods dramatically outperform classical methods. RAFT (Recurrent All-Pairs Field Transforms) is the current benchmark leader for dense optical flow: MDPI Autonomous Control Ch.6

RAFT — 4D Correlation Volume $$\mathbf{C}(i,j,k,l) = \langle \mathbf{f}_1(i,j),\, \mathbf{f}_2(k,l) \rangle$$

$\mathbf{f}_1, \mathbf{f}_2$: feature maps of frame 1 and frame 2 (from shared CNN encoder). The 4D correlation volume measures how similar every pixel in frame 1 is to every pixel in frame 2. A GRU (recurrent unit) iteratively refines the flow estimate by querying this correlation volume, achieving state-of-the-art accuracy (1.5 EPE on Sintel clean) at moderate compute cost (~10 FPS on Jetson Orin). For UAV ego-motion, a lightweight variant (RAFT-Small, 0.9M params) achieves ~30 FPS.

7.6 Monocular Depth Estimation

Plain English First

A single camera image contains depth cues — perspective foreshortening, known object sizes, texture gradients — but extracting metric depth from monocular imagery is an ill-posed problem: infinitely many 3D scenes can produce the same 2D image. Deep learning partially solves this by learning statistical priors about scene geometry from training data. The result is a dense depth map — every pixel gets an estimated distance. For UAVs, this enables terrain reconstruction, obstacle detection, and landing zone assessment without requiring a stereo camera or LiDAR.

Self-Supervised Monocular Depth — The MonoDepth2 Approach

Supervised depth estimation requires LiDAR ground truth — expensive to collect from UAVs. Self-supervised methods use consecutive video frames as supervision: Bestaoui Sebbane §10.3

Self-Supervised Depth Training Loss (MonoDepth2) $$\mathcal{L}_{depth} = \mathcal{L}_{photometric} + \lambda_{smooth}\mathcal{L}_{smooth}$$ $$\mathcal{L}_{photometric} = \min_{t'} \alpha\,\frac{1-\text{SSIM}(I_t, \hat{I}_{t\to t'})}{2} + (1-\alpha)\|I_t - \hat{I}_{t\to t'}\|_1$$

$\hat{I}_{t\to t'}$: frame $t'$ warped to match frame $t$ using predicted depth + estimated camera pose. If depth is correct, warping should perfectly reconstruct frame $t$ from frame $t'$ — the photometric error should be zero. The $\min$ over multiple source frames handles occlusions. $\mathcal{L}_{smooth}$: edge-aware depth smoothness — penalises depth discontinuities unless there is also an image edge there (objects should have smooth depth surfaces). $\alpha = 0.85$ typically.

Scale Ambiguity — The Critical Limitation

Self-supervised monocular depth is scale-ambiguous: the network cannot distinguish between a small object close up and a large object far away from a single image. The predicted depth is up to an unknown scale factor $s$: $\hat{d} = s \cdot d_{metric}$. For UAV navigation, this is critical — obstacle avoidance requires metric distances. Solutions: MDPI Autonomous Control Ch.6

  • IMU integration: Scale is recovered from IMU-estimated displacement between frames (used in MonoDepth2 + IMU fusion)
  • Known object size: If a vehicle of known length is detected, depth can be recovered from apparent size
  • Sparse LiDAR initialisation: Low-density LiDAR provides sparse metric depth points; network learns to propagate to dense
  • Stereo training with known baseline: Scale-aware training with stereo image pairs (baseline = metric depth anchor)

7.7 Edge Deployment — Running DL on UAV Hardware

Plain English First

A model that achieves state-of-the-art accuracy on a GPU server is useless if it cannot run in real time on the 15–50 W compute budget of a UAV payload. Edge deployment is the engineering discipline of making models fast enough, small enough, and energy-efficient enough to run onboard. The tools: quantisation (use 8-bit integers instead of 32-bit floats), pruning (remove unnecessary connections), knowledge distillation (train a small network to mimic a large one), and hardware-optimised inference engines (TensorRT, ONNX Runtime).

Quantisation — INT8 vs FP16 vs FP32

Post-Training Quantisation — Linear Mapping $$x_{int8} = \text{clip}\!\left(\text{round}\!\left(\frac{x_{fp32}}{s}\right) + z,\; -128,\; 127\right)$$ $$s = \frac{x_{max} - x_{min}}{255}, \quad z = -\text{round}\!\left(\frac{x_{min}}{s}\right)$$

$s$: scale factor (maps float range to int range); $z$: zero point. INT8 quantisation reduces model size by 4× and memory bandwidth by 4× vs FP32, enabling ~2–4× inference speedup on hardware with INT8 tensor cores (Jetson Orin, Neural Processing Units). Accuracy impact: typically <1% mAP drop with calibration-aware quantisation. FP16 (half precision) halves size with negligible accuracy loss — easier to apply, less speedup than INT8. Virell §6.2

def conv2d_naive(x, kernel, stride=1, padding=0):
    """
    2D convolution — naive implementation for clarity.
    x:      (H, W, C_in) input feature map
    kernel: (kH, kW, C_in, C_out) filter weights
    Returns: (H_out, W_out, C_out) output feature map
    MDPI Autonomous Control Ch.5
    """
    H, W, C_in = x.shape
    kH, kW, _, C_out = kernel.shape

    # Add padding
    if padding > 0:
        x = np.pad(x, [(padding, padding), (padding, padding), (0, 0)])

    H_out = (H + 2*padding - kH) // stride + 1
    W_out = (W + 2*padding - kW) // stride + 1
    out   = np.zeros((H_out, W_out, C_out))

    for i in range(H_out):
        for j in range(W_out):
            h_start = i * stride; h_end = h_start + kH
            w_start = j * stride; w_end = w_start + kW
            patch = x[h_start:h_end, w_start:w_end, :]   # (kH, kW, C_in)
            # Vectorised over output channels
            out[i, j, :] = np.tensordot(patch, kernel, axes=([0,1,2], [0,1,2]))

    return out


def relu(x): return np.maximum(0, x)


def maxpool2d(x, pool_size=2, stride=2):
    """2×2 max pooling — halves spatial dimensions."""
    H, W, C = x.shape
    H_out = (H - pool_size) // stride + 1
    W_out = (W - pool_size) // stride + 1
    out = np.zeros((H_out, W_out, C))
    for i in range(H_out):
        for j in range(W_out):
            out[i, j, :] = x[i*stride:i*stride+pool_size, j*stride:j*stride+pool_size, :].max(axis=(0,1))
    return out


def batch_norm(x, gamma=None, beta=None, eps=1e-5):
    """
    Batch normalisation — normalises per channel.
    x: (H, W, C) feature map
    Returns normalised map with learned scale gamma and shift beta.
    """
    mu    = x.mean(axis=(0,1), keepdims=True)     # mean per channel
    sigma = x.std(axis=(0,1),  keepdims=True) + eps # std per channel
    x_hat = (x - mu) / sigma
    if gamma is None: gamma = np.ones(x.shape[-1])
    if beta  is None: beta  = np.zeros(x.shape[-1])
    return gamma * x_hat + beta


def count_conv_params(C_in, C_out, K, bias=True):
    """
    Count learnable parameters in a conv layer.
    Formula: (K^2 * C_in + bias) * C_out
    """
    return (K**2 * C_in + 1 if bias else 0) * C_out

    # Conv layer 2: 16→32 channels, 3×3 kernel
    k2 = np.random.randn(3, 3, 16, 32) * 0.1
    out2 = relu(conv2d_naive(out1_pool, k2, stride=1, padding=1))
    print(f"  After Conv2+ReLU: {out2.shape}  params={count_conv_params(16,32,3):,}")

# PART 2 — YOLO Bounding Box Decoder
# Converts raw YOLO output tensor to (x1,y1,x2,y2,conf,class_id)
# Bestaoui Sebbane §9.6
# ════════════════════════════════════════════════════════════════

def decode_yolo_output(raw_output, anchors, n_classes, img_size, conf_thresh=0.5):
    """
    Decode YOLO raw prediction tensor to bounding boxes.

    raw_output: (G, G, A*(5+C)) — grid predictions
    anchors: list of (pw, ph) anchor dimensions (in pixels, relative to img_size)
    n_classes: number of object classes
    img_size: input image size (square assumed)
    conf_thresh: minimum objectness confidence to keep

    Returns: list of [x1, y1, x2, y2, confidence, class_id]
    """
    G = raw_output.shape[0]   # grid size
    A = len(anchors)             # number of anchors
    stride = img_size / G        # pixels per grid cell

    raw = raw_output.reshape(G, G, A, 5 + n_classes)

    detections = []

    for cy in range(G):
        for cx in range(G):
            for a, (pw, ph) in enumerate(anchors):
                pred = raw[cy, cx, a]

                # Decode box centre and size (YOLO parameterisation)
                bx = (1 / (1 + np.exp(-pred[0])) + cx) * stride   # σ(tx) + cx
                by = (1 / (1 + np.exp(-pred[1])) + cy) * stride   # σ(ty) + cy
                bw = pw * np.exp(pred[2])                           # pw * e^tw
                bh = ph * np.exp(pred[3])                           # ph * e^th

                # Objectness + class probability
                obj_conf = 1 / (1 + np.exp(-pred[4]))   # sigmoid
                cls_logits = pred[5:]
                cls_probs  = np.exp(cls_logits) / np.exp(cls_logits).sum()   # softmax
                cls_id     = np.argmax(cls_probs)
                confidence = obj_conf * cls_probs[cls_id]

                if confidence > conf_thresh:
                    # Convert to corner format (x1,y1,x2,y2)
                    x1 = bx - bw/2; y1 = by - bh/2
                    x2 = bx + bw/2; y2 = by + bh/2
                    detections.append([x1, y1, x2, y2, confidence, cls_id])

    return detections

# PART 3 — Non-Maximum Suppression (NMS)
# Removes duplicate detections of the same object
# ════════════════════════════════════════════════════════════════

def compute_iou(boxA, boxB):
    """
    Intersection-over-Union between two boxes [x1,y1,x2,y2].
    IoU = intersection_area / union_area
    """
    xA = max(boxA[0], boxB[0]); yA = max(boxA[1], boxB[1])
    xB = min(boxA[2], boxB[2]); yB = min(boxA[3], boxB[3])
    inter = max(0, xB - xA) * max(0, yB - yA)
    areaA = (boxA[2]-boxA[0]) * (boxA[3]-boxA[1])
    areaB = (boxB[2]-boxB[0]) * (boxB[3]-boxB[1])
    union = areaA + areaB - inter
    return inter / (union + 1e-6)


def non_maximum_suppression(detections, iou_thresh=0.45):
    """
    NMS: suppress overlapping boxes, keep highest confidence.
    detections: list of [x1,y1,x2,y2,conf,class_id]
    Returns: filtered detections
    """
    if not detections: return []
    dets = sorted(detections, key=lambda d: d[4], reverse=True)  # sort by confidence
    kept = []
    while dets:
        best = dets.pop(0)
        kept.append(best)
        # Remove boxes that overlap too much with best
        dets = [d for d in dets if
                d[5] != best[5] or   # different class — always keep
                compute_iou(best[:4], d[:4]) < iou_thresh]
    return kept
