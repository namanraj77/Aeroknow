Semantic SLAM

A drone that knows where it is but not what it is looking at is blind to the world's meaning. Semantic SLAM fuses the geometry of classical SLAM — where am I, what does the structure around me look like — with the perception of deep learning — what are these structures, and what do they imply? The result is a map that answers not just "there is a wall here" but "there is a building here, it is probably occupied, this is a doorway, that is a vehicle." This is the enabling technology for truly autonomous operation in complex, unstructured environments.

11.1 Why Semantic SLAM? The Gap Classical SLAM Leaves Open

Plain English First

Classical SLAM — the kind in Module 04 and Module 05 — builds a geometric map: a point cloud, a voxel grid, or a set of feature landmarks with XYZ coordinates. This map tells the drone where things are. It does not tell the drone what those things are. A point cloud of a building looks exactly like a point cloud of a cliff face. A voxel map of a car park looks exactly like a voxel map of a flat roof. Semantic SLAM attaches meaning to geometry — each region of the map is labelled with an object class, and each object class carries domain knowledge: buildings have entrances, vehicles can move, trees are not structural obstacles, humans require special consideration. This is the difference between a robot that navigates and a robot that understands its environment.

Analogy — The Difference Between a Surveyor and a Soldier

A surveyor maps a city with extreme geometric precision — every building's footprint, every road width, every elevation. But the surveyor's map does not tell you which buildings are hospitals (protected), which roads can support heavy vehicles, which structures are likely to collapse under stress. A soldier interpreting the same city has semantic knowledge overlaid on geometry: "that large open area is a parade ground — likely observed," "that cluster of low buildings is residential — non-combatants likely." Semantic SLAM gives a UAV the soldier's understanding, not just the surveyor's precision.

Real-world autonomous UAV operation requires all three levels simultaneously. The SLAM community has converged on this view — the "SLAM dunk" paper by Cadena et al. (2016) defines the goal as a metric-semantic world model that supports all tasks the robot must perform. MDPI Autonomous Control Ch.6

11.2 Geometric SLAM — The Foundation

Plain English First

Before semantics can be added, you need to understand what classical geometric SLAM actually optimises. The core problem is: given a sequence of sensor measurements and controls, find the most likely trajectory of the robot and the most likely positions of the landmarks in the environment, jointly. This is not two separate problems — they are coupled. Your position estimate depends on where you think the landmarks are; your landmark estimate depends on where you think you were when you observed them.

The SLAM Posterior

The formal SLAM problem estimates the joint posterior over robot poses $\mathbf{x}_{1:T}$ and landmark map $\mathbf{m}$ given all measurements $\mathbf{z}_{1:T}$ and controls $\mathbf{u}_{1:T}$: Bestaoui Sebbane §12.1

Full SLAM Posterior $$p(\mathbf{x}_{1:T},\, \mathbf{m} \mid \mathbf{z}_{1:T},\, \mathbf{u}_{1:T})$$

$\mathbf{x}_{1:T} = \{x_1, x_2, \ldots, x_T\}$: full robot trajectory (all poses). $\mathbf{m} = \{m_1, m_2, \ldots, m_N\}$: map (all landmark positions). $\mathbf{z}_{1:T}$: all measurements (images, LiDAR scans, IMU). $\mathbf{u}_{1:T}$: all controls (velocity commands, odometry). The full SLAM problem keeps the entire trajectory in memory — necessary for global consistency but expensive. Online SLAM marginalises over past poses and only maintains the current pose.

Maximum A Posteriori (MAP) Estimation

Most modern SLAM systems solve the MAP estimate — the trajectory and map that maximises the posterior — rather than maintaining the full distribution: Bestaoui Sebbane §12.1

MAP-SLAM as a Nonlinear Least-Squares Problem $$(\mathbf{x}^*, \mathbf{m}^*) = \arg\min_{\mathbf{x},\mathbf{m}} \sum_{t} \|\mathbf{z}_t - h(\mathbf{x}_t, \mathbf{m})\|^2_{\Sigma_z} + \sum_t \|\mathbf{x}_t - f(\mathbf{x}_{t-1}, \mathbf{u}_t)\|^2_{\Sigma_u}$$

$h(\mathbf{x}_t, \mathbf{m})$: measurement model — predicted observation of landmark $\mathbf{m}$ from pose $\mathbf{x}_t$. $f(\mathbf{x}_{t-1}, \mathbf{u}_t)$: motion model. $\|\cdot\|^2_{\Sigma}$: Mahalanobis distance weighted by covariance $\Sigma$. This is a nonlinear least squares problem solved by Gauss-Newton or Levenberg-Marquardt iteration. Graph-based SLAM (Section 11.4) is the dominant modern approach to solving this efficiently.

11.3 Adding Semantics — What Labels Enable

Plain English First

There are three ways to add semantics to a SLAM map, and they operate at different levels of the pipeline. The simplest paints semantic labels onto an already-built geometric map — a post-processing step. The more powerful approach integrates semantic predictions into the SLAM optimisation itself — labels improve geometry, and geometry improves labels. The most powerful approach represents each object as a unified geometric-semantic entity in the map, so the map contains "car at this pose with this shape" rather than "points at these coordinates" labelled "car."

Three Integration Architectures

Post-hoc Labelling

Run geometric SLAM to completion, then run a semantic segmentation network over keyframes and project labels into the 3D map. Fast. Robust. But semantics don't help SLAM. Used in: SemanticFusion, Kimera (basic mode).

Coupled Optimisation

Include semantic predictions as factors in the SLAM graph — e.g., "two observations labelled 'door' at similar positions are more likely to be the same door than two observations labelled differently." Labels constrain data association and loop closure. Used in: Kimera, SuMa++.

Object-Level SLAM

Replace point landmarks with object landmarks — each map element is a detected object with a pose, a shape model, and a class label. The SLAM state vector includes object poses. Most expressive; most computationally demanding. Used in: EAO-SLAM, CubeSLAM, NodeSLAM.

Neural Implicit Maps

Represent the scene as a continuous neural field (NeRF or 3D Gaussian Splatting) that encodes geometry, appearance, and semantics jointly. Supports novel view synthesis and semantic queries. Emerging; very high quality; slow to train. Used in: Semantic-NeRF, LERF, LangSplat.

Semantic Representation Choices

How semantics are represented in the map determines what questions the map can answer: Marqués & Da Ronch Ch.8

11.4 Factor Graphs — The Unified SLAM Optimisation Framework

Plain English First

A factor graph is a way of writing down a joint probability distribution as a product of simpler functions — each function (called a factor) involves only a small subset of the variables. In SLAM, every pose estimate, every landmark observation, every odometry measurement, and every semantic prediction becomes a factor. The entire SLAM problem — including semantics — is represented as one graph, and optimised jointly. This framework is what allows geometric constraints (IMU, LiDAR scan match) and semantic constraints (two observations of the same object class at nearby positions) to be optimised together in the same system.

Factor Graph Definition

Factor Graph Factorisation of the SLAM Posterior $$p(\mathbf{X}, \mathbf{L} \mid \mathbf{Z}) \propto \prod_i \phi_i(\mathcal{X}_i)$$

$\mathbf{X} = \{x_1, \ldots, x_T\}$: robot pose variables. $\mathbf{L} = \{l_1, \ldots, l_N\}$: landmark/object variables (geometric + semantic). $\phi_i(\mathcal{X}_i)$: a factor — a non-negative function over a small variable subset $\mathcal{X}_i$. The graph has two node types: variable nodes (circles — poses, landmarks) and factor nodes (squares — measurements, constraints). An edge connects a factor to each variable it involves. The MAP estimate minimises the sum of negative log-factor values — a sparse nonlinear least squares problem exploitable by Cholesky factorisation.

Factor Types in Semantic SLAM

Factor TypeVariables InvolvedResidualSource
Odometry factor $(x_t, x_{t+1})$ $\|x_{t+1} - f(x_t, u_t)\|^2_{\Sigma_u}$ IMU preintegration, wheel odometry
Landmark observation factor $(x_t, l_j)$ $\|z_{tj} - h(x_t, l_j)\|^2_{\Sigma_z}$ Feature point, LiDAR point
Loop closure factor $(x_i, x_j)$ $\|x_j - T_{ij} x_i\|^2_{\Sigma_{lc}}$ Place recognition (geometric or semantic)
Semantic observation factor $(x_t, l_j^{sem})$ $-\log p(c_{tj} \mid \text{class}(l_j))$ Object detector output (class + confidence)
Object shape factor $(l_j^{pose}, l_j^{shape})$ $\|p_k - s(l_j^{pose}, l_j^{shape})\|^2$ Shape model (ellipsoid, CAD, learned)
Semantic consistency factor $(l_i, l_j)$ $-\log p(\text{adjacent} \mid \text{class}(l_i), \text{class}(l_j))$ Scene ontology (cars near roads, not rooftops)
Theorem 11.1 — Information Matrix Sparsity

The Hessian (information matrix) $\mathbf{H}$ of the MAP-SLAM least squares problem has the same sparsity pattern as the factor graph adjacency structure:

$$\mathbf{H}_{ij} \neq 0 \quad \Leftrightarrow \quad \exists \text{ factor } \phi_k \text{ connecting variable } i \text{ and variable } j$$

For a robot visiting $T$ poses and observing $N$ landmarks where each pose observes $k \ll N$ landmarks: $\mathbf{H}$ is $(6T + 3N) \times (6T + 3N)$ but with only $O(Tk)$ non-zero off-diagonal blocks. Cholesky factorisation of a sparse matrix costs $O(n \cdot s^2)$ where $s$ is the fill-in — far cheaper than the $O(n^3)$ cost of dense factorisation. This sparsity is what makes graph SLAM tractable for large maps: a 10,000-pose, 100,000-landmark map with $k=5$ is solved in seconds on a CPU. Adding semantic factors increases $N$ (each object is a new variable) but the sparsity structure is maintained — semantic SLAM is as tractable as geometric SLAM when factors are local. Bestaoui Sebbane §12.2

Worked Example 11.1 — Building the Factor Graph for a 3-Pose UAV Mission

A UAV flies 3 poses ($x_1, x_2, x_3$). It observes geometric landmark $l_1$ from $x_1$ and $x_2$, and detects object $o_1$ (a vehicle) from $x_2$ and $x_3$. Count variables, factors, and the Hessian dimension.

Variables: $x_1, x_2, x_3$ (poses, 6-DOF each) + $l_1$ (geometric landmark, 3D) + $o_1^{pos}$ (object position, 3D) + $o_1^{sem}$ (object class, categorical) = 3×6 + 3 + 3 + 1 = 25 variables.

Factors: (1) Prior on $x_1$ — 1 factor. (2) Odometry $(x_1,x_2)$, $(x_2,x_3)$ — 2 factors. (3) Landmark observation $(x_1,l_1)$, $(x_2,l_1)$ — 2 factors. (4) Object observation $(x_2,o_1^{pos})$, $(x_3,o_1^{pos})$ — 2 factors. (5) Semantic factor $(x_2,o_1^{sem})$, $(x_3,o_1^{sem})$ — 2 factors. Total: 9 factors.

Hessian: 25×25 = 625 entries total. Non-zero blocks: only variables connected by a factor have non-zero cross-terms. Sparsity ≈ 85% — most of the matrix is zero. This is what enables real-time optimisation. Bestaoui Sebbane §12.2

11.5 Object-Level SLAM — Landmarks with Meaning

Plain English First

Instead of tracking thousands of anonymous feature points, object-level SLAM tracks a small number of named objects — "car #3", "building entrance #7", "barrel #12". Each object has a pose (6-DOF position and orientation), a shape (parameterised by an ellipsoid, a bounding box, or a learned latent code), and a semantic class (vehicle, structure, landmark). The drone observes these objects across many frames, and the SLAM optimiser simultaneously refines where the drone was when it saw them and where the objects actually are. The resulting map is compact (tens of objects instead of millions of points), semantically queryable ("find all vehicles"), and robust to dynamic objects (moving cars can be excluded from the map by their class label).

Object Representation: The Ellipsoidal Model

The simplest object shape model that supports 3D SLAM is the ellipsoid: a second-order surface parametrised by a 4×4 dual quadric matrix. Marqués & Da Ronch Ch.8

Dual Quadric Object Representation $$\mathcal{Q}^* = \begin{pmatrix} \mathbf{U}\mathbf{U}^\top + \mathbf{t}\mathbf{t}^\top - \mathbf{t}\mathbf{t}^\top & \mathbf{t} \\ \mathbf{t}^\top & -1 \end{pmatrix}$$

$\mathbf{U} = \text{diag}(a, b, c)$: semi-axes of the ellipsoid. $\mathbf{t} \in \mathbb{R}^3$: centre position. A 2D bounding box observation $\mathbf{B}$ from camera pose $\mathbf{P}$ constrains $\mathcal{Q}^*$ via the projection: $\mathbf{C}^* = \mathbf{P}\,\mathcal{Q}^*\,\mathbf{P}^\top$ where $\mathbf{C}^*$ is the dual conic of the projected ellipse, and the bounding box provides 4 linear constraints on $\mathbf{C}^*$. Three non-degenerate views from different camera poses fully constrain the ellipsoid (9 DOF = semi-axes 3 + position 3 + orientation 3).

Data Association — The Hard Problem

Before optimising object poses, each new detection must be associated with an existing map object or spawned as a new one. Wrong associations corrupt the map. Three approaches of increasing sophistication: MDPI Autonomous Control Ch.6

MethodAssociation CriterionProsCons
Nearest Neighbour IoU of projected bounding boxes + class match Fast, simple Fails when objects overlap or are briefly occluded
Hungarian (optimal) Minimum cost bipartite matching on IoU + distance + class Globally optimal for current frame $O(N^3)$; assumes detections are independent
Probabilistic (JCBB) Joint compatibility branch-and-bound on Mahalanobis distance Correct under uncertainty; handles ambiguity Exponential worst case; needs covariance estimates
Theorem 11.2 — Data Association as Maximum Likelihood

The optimal data association $\mathbf{a}^*$ between detections $\mathbf{d}_{1:K}$ and existing map objects $\mathbf{o}_{1:M}$ maximises the joint likelihood:

$$\mathbf{a}^* = \arg\max_{\mathbf{a}} \prod_{k=1}^K p(\mathbf{d}_k \mid \mathbf{o}_{a(k)}, \mathbf{x}_t) \cdot p(\mathbf{a} \mid \mathbf{d}_{1:K})$$

$p(\mathbf{d}_k \mid \mathbf{o}_{a(k)}, \mathbf{x}_t)$: likelihood of detection $k$ given object $a(k)$ at pose $\mathbf{x}_t$ — depends on IoU, semantic class match, and distance in the projected ellipsoid. $p(\mathbf{a})$: prior penalising many new object hypotheses (Occam's razor). This MAP assignment is NP-hard in general — practical systems use Hungarian or greedy nearest-neighbour with a gating threshold (reject associations where Mahalanobis distance $> \chi^2_{0.99}$ for the observation dimension). Marqués & Da Ronch Ch.8

Worked Example 11.2 — Object SLAM Data Association: Vehicle Tracking

A UAV at altitude 30 m detects 3 vehicles across two consecutive frames. Frame 1: detections D1, D2 at pixel positions (320,240), (480,360). Frame 2 (drone has moved 5 m east): D3, D4, D5 at (290,220), (440,330), (600,280). Apply IoU + class-based Hungarian assignment. All detections are class "car."

Step 1 — Project map objects to frame 2: Using the drone's odometry (5 m east), project the two existing objects O1 (from D1) and O2 (from D2) into frame 2's camera. Projected positions: O1→(285,218), O2→(435,328).

Step 2 — Build IoU cost matrix: IoU between projected bounding boxes and new detections. Assume 50×50 px bounding boxes. C[D3,O1] = 0.82 (high overlap), C[D4,O2] = 0.79, C[D5,O1] = 0.12, C[D5,O2] = 0.15.

Step 3 — Hungarian assignment: Maximise total IoU → D3→O1, D4→O2 (confirmed), D5 → new object O3 (IoU with any existing < 0.3 threshold).

Result: Two existing vehicles tracked, one new vehicle spawned. The new vehicle O3 gets an initial ellipsoid from the pixel bounding box back-projected to 3D using altitude 30 m + camera intrinsics. After 3 views of O3 from different angles, the ellipsoid is fully constrained. MDPI Autonomous Control Ch.6

11.6 Neural Implicit Representations — NeRF and 3D Gaussian Splatting

Plain English First

Neural Radiance Fields (NeRF) and 3D Gaussian Splatting (3DGS) represent a completely different paradigm for scene reconstruction. Instead of building an explicit point cloud or mesh, they train a neural network (or a set of Gaussian primitives) to encode the entire scene's appearance and geometry. The network takes a 3D position and a viewing direction as input and outputs colour and density. To render an image, you query the network along rays from the camera. To add semantics, you extend the output to include class labels — every point in 3D space now carries both appearance and semantic meaning. The result supports photorealistic novel view synthesis and semantic queries, making it the most expressive scene representation ever built.

Neural Radiance Fields (NeRF)

NeRF represents the scene as a continuous function approximated by an MLP: Marqués & Da Ronch Ch.8

NeRF Radiance Field — Scene Function $$F_\theta: (\mathbf{x}, \mathbf{d}) \mapsto (\mathbf{c}, \sigma)$$ $$\mathbf{x} \in \mathbb{R}^3 \text{ (3D position)}, \quad \mathbf{d} \in \mathbb{S}^2 \text{ (viewing direction)}, \quad \mathbf{c} \in [0,1]^3 \text{ (RGB colour)}, \quad \sigma \in \mathbb{R}_{\geq 0} \text{ (volume density)}$$

Volume rendering: the colour of a ray $\mathbf{r}(t) = \mathbf{o} + t\mathbf{d}$ is computed by integrating along the ray: $\hat{\mathbf{C}}(\mathbf{r}) = \int_{t_n}^{t_f} T(t)\,\sigma(\mathbf{r}(t))\,\mathbf{c}(\mathbf{r}(t), \mathbf{d})\,dt$ where $T(t) = \exp\!\left(-\int_{t_n}^t \sigma(\mathbf{r}(s))\,ds\right)$ is the accumulated transmittance. Training minimises the photometric loss: $\mathcal{L} = \sum_{\mathbf{r}} \|\hat{\mathbf{C}}(\mathbf{r}) - \mathbf{C}(\mathbf{r})\|^2$.

Semantic NeRF — Adding Labels to the Field

Semantic NeRF extends the output to include a class label per point: Marqués & Da Ronch Ch.8

Semantic NeRF Output Extension $$F_\theta^{sem}: (\mathbf{x}, \mathbf{d}) \mapsto (\mathbf{c}, \sigma, \mathbf{s})$$

$\mathbf{s} \in \mathbb{R}^C$: logit vector over $C$ semantic classes. The semantic label of a ray is rendered by the same volume integral: $\hat{\mathbf{S}}(\mathbf{r}) = \int_{t_n}^{t_f} T(t)\,\sigma(\mathbf{r}(t))\,\mathbf{s}(\mathbf{r}(t))\,dt$. Supervision: cross-entropy loss on predicted vs. ground-truth 2D semantic segmentation. The network learns to associate geometry (density $\sigma$) with semantics ($\mathbf{s}$) — a wall point carries "building" label; a tree point carries "vegetation." Semantic consistency across views comes for free — the 3D field inherently propagates labels from labelled views to unlabelled viewpoints.

3D Gaussian Splatting (3DGS) — Real-Time Alternative

NeRF is photorealistic but slow to render — 30 s per frame at full resolution. 3DGS achieves real-time rendering by replacing the implicit MLP with explicit anisotropic Gaussian primitives:

3DGS Gaussian Primitive $$\mathcal{G}_k = \{\boldsymbol{\mu}_k,\, \mathbf{\Sigma}_k,\, \alpha_k,\, \mathbf{c}_k\}$$ $$\mathbf{\Sigma}_k = \mathbf{R}_k \mathbf{S}_k \mathbf{S}_k^\top \mathbf{R}_k^\top, \quad \mathbf{S}_k = \text{diag}(s_x, s_y, s_z)$$

$\boldsymbol{\mu}_k \in \mathbb{R}^3$: Gaussian centre. $\mathbf{\Sigma}_k$: covariance (shape and orientation). $\alpha_k \in [0,1]$: opacity. $\mathbf{c}_k$: view-dependent colour (spherical harmonics). Rendering: sort Gaussians by depth, splat each into image plane via 2D projection, alpha-composite front-to-back. GPU rasterisation: 100+ FPS at HD resolution. Semantic 3DGS (LangSplat, SAGA) adds a language feature vector $\mathbf{f}_k \in \mathbb{R}^d$ per Gaussian, enabling text-queried semantic search in real time.

11.7 Semantic Loop Closure — Closing the Map Correctly

Plain English First

Loop closure is when the robot recognises that it has returned to a previously visited location. This recognition allows the SLAM system to constrain all the accumulated drift in the trajectory — suddenly the map "snaps" into global consistency. Without loop closure, drift is unbounded; with it, long-duration missions can maintain centimetre-level accuracy. Classical loop closure uses visual appearance (bag-of-words descriptors like DBoW). Semantic loop closure is more powerful: "I recognise this place because it has a building with a red door, a parked van, and a junction" — semantic landmarks are more distinctive than geometric features and more robust to lighting changes and seasonal variation.

Semantic Descriptor for Loop Closure

A simple but effective semantic place descriptor counts the histogram of object classes observed from each viewpoint: MDPI Autonomous Control Ch.6

Semantic Histogram Descriptor — Place Representation $$\mathbf{h}(x_t) = \frac{1}{K} \sum_{k=1}^{K} \mathbf{e}_{c_k} \in \mathbb{R}^C, \quad \|\mathbf{h}\|_1 = 1$$ $$\text{similarity}(x_i, x_j) = \cos(\mathbf{h}(x_i), \mathbf{h}(x_j)) = \frac{\mathbf{h}(x_i) \cdot \mathbf{h}(x_j)}{\|\mathbf{h}(x_i)\|\,\|\mathbf{h}(x_j)\|}$$

$K$: number of detected objects in view at pose $x_t$. $\mathbf{e}_{c_k}$: one-hot class vector for object $k$. $\mathbf{h}(x_t)$: normalised class frequency histogram. A loop closure candidate is proposed when cosine similarity $> \tau$ (typically 0.85). Advantage over BoW: completely invariant to illumination and texture. Weakness: ambiguous in scenes with uniform class distribution (all-grass field, featureless warehouse). Extension: weight by object rarity — $w_{c_k} \propto -\log p(c_k)$ (IDF weighting, analogous to TF-IDF in text retrieval).

11.8 Semantic SLAM for Military UAV Operations

Plain English First

Semantic SLAM is not an academic exercise for military UAVs — it is the difference between a platform that can follow a pre-programmed route and one that can adapt to a dynamic battlespace. A drone building a semantic map knows which structures are occupied, which vehicles are military vs civilian, which areas have changed since the last pass, and which targets have been confirmed vs suspected. It can use that knowledge to replanning missions, adjusting threat assessments, and communicating actionable intelligence to human operators — not just raw sensor data.

    def pose_slice(self, i: int): return slice(i*3, i*3+3)
    def lmk_slice(self,  j: int): return slice(self.n_poses*3 + j*2,
                                              self.n_poses*3 + j*2+2)

    def add_odometry(self, i: int, j: int, Δx: float, Δy: float, Δθ: float,
                      σ_trans=0.1, σ_rot=0.05):
        """Add odometry factor between poses i and j."""
        z    = np.array([Δx, Δy, Δθ])
        Σ_inv= np.diag([1/σ_trans**2, 1/σ_trans**2, 1/σ_rot**2])

        def residual(state):
            xi = state[self.pose_slice(i)]
            xj = state[self.pose_slice(j)]
            # Relative pose in frame i
            cθ, sθ = np.cos(xi[2]), np.sin(xi[2])
            R = np.array([[cθ, sθ], [-sθ, cθ]])
            Δpos = R @ (xj[:2] - xi[:2])
            Δrot = xj[2] - xi[2]
            return np.array([Δpos[0]-z[0], Δpos[1]-z[1], Δrot-z[2]])

        self.factors.append(('odom', (i, j), residual, Σ_inv))

    def add_landmark_obs(self, i: int, j: int, rx: float, ry: float,
                           σ_obs=0.2):
        """Add landmark observation factor: pose i observes landmark j at (rx,ry) in body frame."""
        z    = np.array([rx, ry])
        Σ_inv= np.eye(2) / σ_obs**2

        def residual(state):
            xi = state[self.pose_slice(i)]
            lj = state[self.lmk_slice(j)]
            cθ, sθ = np.cos(xi[2]), np.sin(xi[2])
            R = np.array([[cθ, sθ], [-sθ, cθ]])
            pred = R @ (lj - xi[:2])
            return pred - z

        self.factors.append(('lmk', (i, j), residual, Σ_inv))

    def add_semantic_factor(self, i: int, j: int, k: int,
                             class_match: bool, σ_sem=0.3):
        """
        Semantic consistency factor: landmarks j and k are the same class.
        If class_match=True: they should be close (same semantic object).
        If class_match=False: weak repulsion (different class, shouldn't overlap).
        Bestaoui Sebbane §12.2
        """
        sign = -1.0 if class_match else 1.0   # attract or repel
        Σ_inv= np.eye(2) / σ_sem**2

        def residual(state):
            lj = state[self.lmk_slice(j)]
            lk = state[self.lmk_slice(k)]
            return sign * (lk - lj)

        self.factors.append(('sem', (j, k), residual, Σ_inv))

    def total_cost(self) -> float:
        cost = 0.0
        for _, _, res_fn, Σ_inv in self.factors:
            r = res_fn(self.state)
            cost += float(r @ Σ_inv @ r)
        return cost

    def optimise(self, n_iter: int = 20, lr: float = 0.5) -> list:
        """Gauss-Newton with numerical Jacobian (for clarity; production uses analytical)."""
        costs = []
        eps   = 1e-5
        # Fix first pose (anchor — removes gauge freedom)
        fixed = list(range(3))

        for it in range(n_iter):
            H = np.zeros((self.n_vars, self.n_vars))
            g = np.zeros(self.n_vars)

            for _, indices, res_fn, Σ_inv in self.factors:
                r = res_fn(self.state)
                m = len(r)
                # Numerical Jacobian
                J = np.zeros((m, self.n_vars))
                for k in range(self.n_vars):
                    s_plus  = self.state.copy(); s_plus[k]  += eps
                    s_minus = self.state.copy(); s_minus[k] -= eps
                    J[:, k] = (res_fn(s_plus) - res_fn(s_minus)) / (2*eps)

                H += J.T @ Σ_inv @ J
                g += J.T @ Σ_inv @ r

            # Anchor: fix first pose by removing from system
            for idx in fixed:
                H[idx, :] = 0; H[:, idx] = 0; H[idx, idx] = 1
                g[idx] = 0

            try:
                Δ = np.linalg.solve(H, -g)
            except np.linalg.LinAlgError:
                break

            self.state += lr * Δ
            c = self.total_cost()
            costs.append(c)

            if len(costs) > 2 and abs(costs[-1] - costs[-2]) < 1e-6:
                break

        return costs

class SemanticPlaceDescriptor:
    """
    Class-histogram place descriptor with IDF weighting.
    MDPI Autonomous Control Ch.6
    """

    def __init__(self, class_names: list, idf_weight: bool = True):
        self.class_names = class_names
        self.C           = len(class_names)
        self.class_idx   = {c: i for i, c in enumerate(class_names)}
        self.idf_weight  = idf_weight
        self._desc_db    = {}       # pose_id → descriptor vector
        self._class_freq = np.zeros(self.C)   # for IDF computation
        self._n_poses    = 0

    def add_observation(self, pose_id: int, detections: list):
        """
        detections: list of class name strings seen from this pose.
        Compute normalised class histogram → store as descriptor.
        """
        hist = np.zeros(self.C)
        for cls in detections:
            idx = self.class_idx.get(cls)
            if idx is not None:
                hist[idx] += 1
                self._class_freq[idx] += 1
        self._n_poses += 1

        if hist.sum() > 0:
            hist /= hist.sum()   # L1 normalise

        self._desc_db[pose_id] = hist

    def _idf_weights(self) -> np.ndarray:
        """IDF: rare classes get higher weight. -log(class_freq / n_poses)."""
        freq = np.maximum(self._class_freq, 1) / max(self._n_poses, 1)
        return -np.log(freq + 1e-9)

    def find_loop_closures(self, query_id: int, threshold: float = 0.85,
                            min_gap: int = 10) -> list:
        """
        Find all past poses similar to query_id.
        Returns list of (pose_id, similarity) sorted by similarity descending.
        min_gap: minimum index distance to avoid short-term revisits.
        """
        if query_id not in self._desc_db:
            return []

        q = self._desc_db[query_id].copy()
        idf = self._idf_weights() if self.idf_weight else np.ones(self.C)

        # Weight by IDF
        q_w = q * idf

        candidates = []
        for pid, hist in self._desc_db.items():
            if abs(pid - query_id) < min_gap: continue
            h_w = hist * idf
            nq  = np.linalg.norm(q_w); nh = np.linalg.norm(h_w)
            if nq < 1e-8 or nh < 1e-8: continue
            sim = float(q_w @ h_w) / (nq * nh)
            if sim >= threshold:
                candidates.append((pid, sim))

        return sorted(candidates, key=lambda x: -x[1])

# PART 3 — Object Data Association (Hungarian + IoU gating)
# Marqués & Da Ronch Ch.8, Theorem 11.2
# ════════════════════════════════════════════════════════════════

def box_iou(b1, b2) -> float:
    """IoU between two [x_centre, y_centre, w, h] boxes."""
    x1m, y1m = b1[0]-b1[2]/2, b1[1]-b1[3]/2
    x1M, y1M = b1[0]+b1[2]/2, b1[1]+b1[3]/2
    x2m, y2m = b2[0]-b2[2]/2, b2[1]-b2[3]/2
    x2M, y2M = b2[0]+b2[2]/2, b2[1]+b2[3]/2
    inter_w = max(0, min(x1M,x2M) - max(x1m,x2m))
    inter_h = max(0, min(y1M,y2M) - max(y1m,y2m))
    inter   = inter_w * inter_h
    union   = b1[2]*b1[3] + b2[2]*b2[3] - inter
    return inter / union if union > 0 else 0


def associate_objects(detections: list, map_objects: list,
                        iou_threshold: float = 0.3,
                        class_match_required: bool = True) -> dict:
    """
    Hungarian-based data association between new detections and existing map objects.

    detections:   list of {'box': [cx,cy,w,h], 'class': str, 'conf': float}
    map_objects:  list of {'id': int, 'proj_box': [cx,cy,w,h], 'class': str}
    Returns:      dict {detection_idx → map_object_id or None (new object)}

    Marqués & Da Ronch Ch.8, Theorem 11.2
    """
    D, M = len(detections), len(map_objects)
    assignment = {i: None for i in range(D)}

    if D == 0 or M == 0:
        return assignment

    # Build cost matrix: 1 - IoU (Hungarian minimises cost)
    cost = np.ones((D, M))
    for i, det in enumerate(detections):
        for j, obj in enumerate(map_objects):
            # Gate: class must match (if required)
            if class_match_required and det['class'] != obj['class']:
                cost[i, j] = 1.0   # maximum cost → won't be assigned
                continue
            iou = box_iou(det['box'], obj['proj_box'])
            cost[i, j] = 1.0 - iou

    # Hungarian assignment (minimise cost)
    row_ind, col_ind = linear_sum_assignment(cost)

    for r, c in zip(row_ind, col_ind):
        iou = 1.0 - cost[r, c]
        if iou >= iou_threshold:
            assignment[r] = map_objects[c]['id']
        # else: detection r is a new object (assignment[r] stays None)

    return assignment
