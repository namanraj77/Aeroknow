Autonomy & Decision-Making

Knowing where you are (navigation) and what you see (perception) is not enough. A truly autonomous UAV must decide what to do next — in real time, under uncertainty, with incomplete information, against a potentially adaptive adversary, in environments where communication may be denied. This module covers the formal frameworks — POMDP, Behaviour Trees, path planning, OODA loop — that structure autonomous decision-making in military UAV systems.

8.1 Levels of Autonomy — A Spectrum, Not a Switch

Plain English First

Autonomy is not binary — it is a spectrum from "human does everything" to "machine does everything." Military doctrine and system engineering both require being precise about which decisions are made by the machine vs. the human, and under what conditions that boundary shifts. A UAV might autonomously handle attitude stabilisation (Level 1) while requiring human approval for every weapon release (Level 4 authorisation). Understanding this spectrum is both a technical requirement and a legal/ethical one — it determines accountability, certification requirements, and rules of engagement compliance.

LEVEL 1
Remote Control
Human executes every control input. UAV has no independent decision-making. Example: direct RC link.
LEVEL 2
Autopilot
Human sets goals; machine handles low-level execution (attitude, altitude hold). Pilot in command. Example: PX4 stabilised mode.
LEVEL 3
Supervised Auto
Machine executes full missions autonomously; human monitors and can override. Human on the loop. Example: MQ-9 ISR orbit.
LEVEL 4
Goal-Directed
Machine plans and re-plans to achieve objectives; human sets goals and authorises key actions. Example: autonomous strike with MITL engagement.
LEVEL 5
Fully Autonomous
Machine selects and executes all actions including lethal engagement without per-action human authorisation. Legally contested. Example: Harpy NG.

Bestaoui Sebbane maps these levels to specific technical capability requirements, noting that each step up the autonomy ladder requires solving progressively harder problems: Level 2 requires only control theory (Module 03). Level 3 requires navigation + perception (Modules 04–07). Level 4–5 require decision theory, planning under uncertainty, and — controversially — machine ethics. Bestaoui Sebbane §9.1

DoD Directive 3000.09 — Autonomous Weapons Policy US Department of Defense Directive 3000.09 (updated 2023) requires that autonomous and semi-autonomous weapons systems be designed to allow commanders and operators to exercise "appropriate levels of human judgment over the use of force." It does not prohibit LAWS — but requires senior official waivers for systems that cannot meet human judgment requirements. The directive mandates: (1) system testing to detect unintended engagements; (2) mechanisms to pause/stop operations; (3) IHL compliance verification. MDPI Autonomous Control Ch.6

8.2 The OODA Loop — Decision Cycle Framework

Plain English First

Every decision — by a human or a machine — follows a cycle: you observe the situation, you interpret what you observed, you decide what to do, and you act. Then the cycle repeats. Colonel John Boyd called this the OODA loop (Observe-Orient-Decide-Act). The key military insight is: the side that completes this loop faster wins. If your UAV can observe a threat, orient to it, decide to evade, and act — before the adversary can complete their own OODA cycle — you survive. This temporal competition is why autonomous systems are so militarily valuable: they complete the loop in milliseconds, where a human pilot needs seconds.

O
OBSERVE
Sensor data collection. EO/IR, radar, SIGINT, datalink. UAV: runs continuously at sensor rate (30–1000 Hz).
O
ORIENT
Fuse observations into situational awareness. EKF, object detection, threat classification. This is the most complex step — shapes everything else.
D
DECIDE
Select an action from available options. POMDP solver, behaviour tree, mission planner. Constrained by ROE, fuel, capabilities.
A
ACT
Execute the chosen action. GNC controllers (Module 03). Physical actuation. Feedback re-enters the Observe phase immediately.
OODA Loop Cycle Time — Competitive Analysis $$T_{OODA} = T_{obs} + T_{orient} + T_{decide} + T_{act}$$ $$T_{OODA}^{human} \approx 0.25 + 0.5 + 1.0 + 0.25 = 2.0 \text{ s (trained pilot under stress)}$$ $$T_{OODA}^{UAV-auto} \approx 0.03 + 0.05 + 0.01 + 0.01 = 0.10 \text{ s (onboard AI)}$$

Autonomous UAV completes 20 OODA cycles per human OODA cycle. In a kinetic engagement (missile evasion, intercept, drone-vs-drone), this temporal advantage is decisive. The Orient phase dominates autonomous OODA time — it is where sensor fusion, ML perception, and threat classification run. Reducing $T_{orient}$ through faster inference hardware (Module 07 edge deployment) directly improves survivability. Bestaoui Sebbane §9.2

8.3 POMDP — Decision-Making Under Uncertainty

Plain English First

The MDP from Module 06 assumed the UAV always knows exactly what state it is in. Reality is different: you don't know exactly where the enemy is — you have noisy sensor readings. You don't know whether a detected object is a combatant or civilian — you have an imperfect classifier. You don't know if your navigation estimate is accurate — you have a covariance ellipsoid. The POMDP (Partially Observable MDP) generalises the MDP to handle this fundamental uncertainty — making it the right mathematical framework for autonomous military UAV decision-making.

Analogy — Playing Chess Blindfolded

A standard MDP is like playing chess where you can see the full board. A POMDP is like playing chess blindfolded: someone tells you what your opponent's last move probably was (noisy observation), but you must maintain a mental model of where all the pieces might be (belief state) and choose your move accordingly. The optimal POMDP policy chooses actions that both progress toward the goal and gather information that reduces uncertainty — exactly the trade-off a military UAV must make when deciding whether to approach a target for better identification.

POMDP Formalisation

POMDP Tuple — Extension of MDP with Partial Observability $$\mathcal{M}_{POMDP} = (\mathcal{S},\, \mathcal{A},\, \mathcal{O},\, P,\, Z,\, R,\, \gamma,\, b_0)$$

$\mathcal{O}$: observation space (sensor readings, classifier outputs); $Z(o|s',a)$: observation model — probability of observing $o$ in state $s'$ after action $a$; $b_0$: initial belief state — prior probability distribution over possible states. Everything else is the same as the MDP. The key difference: the agent never directly sees $s$ — it only sees observations $o$ and must maintain a belief state $b: \mathcal{S} \to [0,1]$.

Belief State Update — The POMDP Filter

Belief Update (Bayes Filter) $$b'(s') = \eta\, Z(o|s',a) \sum_{s \in \mathcal{S}} P(s'|s,a)\, b(s)$$

$\eta$: normalisation constant; $o$: new observation; $a$: action just taken. This is exactly the Bayesian update from Module 04 and 05, now applied to discrete state spaces. The EKF from Module 04 is a special case of the belief update for continuous Gaussian state spaces. The POMDP is the general framework; the EKF is the computational tractable approximation for navigation. Bestaoui Sebbane §9.3

Value Functions and Policies Over Beliefs

In a POMDP, the policy maps beliefs to actions: $\pi: \Delta(\mathcal{S}) \to \mathcal{A}$. The value function over beliefs is piecewise linear and convex (PWLC): MDPI Autonomous Control Ch.6

Alpha-Vector Representation of POMDP Value Function $$V^*(b) = \max_{\alpha \in \Gamma} \sum_{s} \alpha(s)\, b(s) = \max_{\alpha \in \Gamma}\, \alpha \cdot b$$

$\Gamma$: set of $\alpha$-vectors (one per policy fragment); $\alpha(s)$: expected cumulative reward of a particular action sequence starting in state $s$. The value function is the upper envelope of all $\alpha$-vectors. Computing $\Gamma$ exactly is PSPACE-complete — intractable for large state spaces. Practical algorithms: PBVI (Point-Based Value Iteration), SARSOP, POMCP (Monte Carlo Tree Search). For UAV mission planning with $|\mathcal{S}| < 100$ states, PBVI converges in <1 s. For rich sensor-observation spaces, POMCP is the standard approach.

Worked Example 8.1 — POMDP for Target Identification Under Uncertainty

A UAV must decide whether to engage or not engage a detected target. The target is either a combatant vehicle (state $s_1$) or a civilian vehicle (state $s_2$). The onboard classifier gives: P(classify as threat | combatant) = 0.85, P(classify as threat | civilian) = 0.12. Prior belief: P($s_1$) = 0.4. After observing "threat classification" from the sensor, compute the updated belief and determine whether the POMDP engage threshold (belief $> 0.9$) is met.

Observation model: $Z(\text{threat}|s_1) = 0.85$, $Z(\text{threat}|s_2) = 0.12$. Prior: $b(s_1) = 0.4$, $b(s_2) = 0.6$.

Unnormalised update: $b'(s_1) \propto Z(\text{threat}|s_1) \cdot b(s_1) = 0.85 \times 0.4 = 0.340$ $b'(s_2) \propto Z(\text{threat}|s_2) \cdot b(s_2) = 0.12 \times 0.6 = 0.072$

Normalisation: $\eta = 1/(0.340 + 0.072) = 1/0.412 = 2.427$

Posterior: $b'(s_1) = 0.340 \times 2.427 = \mathbf{0.825}$, $b'(s_2) = 0.072 \times 2.427 = \mathbf{0.175}$

Decision: $b'(s_1) = 0.825 < 0.9$ — threshold not met. Do not engage. Continue observation to reduce uncertainty. A second "threat" classification (same observation) would give: $b''(s_1) \approx 0.85 \times 0.825 / (0.85 \times 0.825 + 0.12 \times 0.175) = 0.701 / 0.722 = \mathbf{0.971} > 0.9$ → engage threshold met.

Key insight: The POMDP naturally implements an "information gathering before action" strategy. The UAV takes multiple observations, accumulating evidence, before crossing the engagement threshold. This is a formal implementation of the IHL precaution principle and is a core argument for POMDP-based autonomy in contested military environments. Bestaoui Sebbane §9.4

8.4 Behaviour Trees — Modular Mission Logic

Plain English First

A Behaviour Tree (BT) is a way to organise a UAV's decision logic as a hierarchical tree of conditions and actions. Think of it like a flowchart, but more structured and much easier to modify, test, and combine. Each "leaf" node is either a condition check ("Is the battery above 20%?") or an action ("Fly to waypoint X"). Internal nodes combine their children using logic operators: "do all of these in sequence" (Sequence), or "try each until one succeeds" (Selector/Fallback). BTs are the dominant mission logic framework in autonomous UAV systems — used in PX4, ROS2 Navigation, and nearly every academic and military UAV system — because they are readable, testable, and composable in a way that finite state machines are not.

BT Node Types and Return Status

Node TypeSymbolLogicReturns
Sequence (→)→Execute children left to right. Stop and return FAILURE if any child fails.SUCCESS if all succeed; FAILURE on first failure
Selector / Fallback (?)?Try each child. Stop and return SUCCESS if any child succeeds.SUCCESS on first success; FAILURE if all fail
Parallel (⇉)⇉Execute all children simultaneously. Success threshold $M/N$ configurable.SUCCESS if $\geq M$ children succeed
Decorator◇Modifies child's return value or execution count (Repeat, Retry, Invert, etc.)Depends on decorator type
Action (□)□Leaf node: executes a task. Returns RUNNING while executing.SUCCESS / FAILURE / RUNNING
Condition (◇)◇Leaf node: checks a predicate. Instantaneous (no RUNNING state).SUCCESS (true) / FAILURE (false)

A Military UAV ISR Mission Behaviour Tree

? [Root Fallback — always tries to do something useful] ├─ → [Emergency Handler Sequence] │ ├─ ◇ Battery < 10% OR Critical Failure │ └─ □ Execute Emergency Return-to-Base ├─ ? [Mission Fallback — try primary then contingency] │ ├─ → [Primary ISR Mission Sequence] │ │ ├─ ◇ GPS / Nav estimate valid │ │ ├─ ◇ Target area reachable (fuel check) │ │ ├─ □ Transit to ISR orbit waypoint │ │ └─ → [Collect Intelligence Loop] │ │ ├─ ◇ Repeat(N=10) [collect 10 frames] │ │ │ └─ □ Capture + process ISR image │ │ └─ □ Transmit data to GCS │ └─ → [Degraded Mode Sequence] │ ├─ ◇ Comms link lost > 60s │ ├─ □ Switch to autonomous loiter │ └─ □ Execute pre-planned fallback waypoints └─ ? Theorem 8.1 — BT Reactivity and Modularity Properties

A Behaviour Tree with $N$ nodes has the following formal properties: MDPI Autonomous Control Ch.7

1. Reactivity: If any condition node changes from SUCCESS to FAILURE, the tree immediately re-routes on the next tick — without explicit state transition code. This emergent reactivity is the key advantage over finite state machines (FSMs), which require explicit transitions between every pair of states: $O(N^2)$ transitions for $N$ states vs. $O(N)$ nodes for a BT of equivalent complexity.

2. Composability: Any subtree is a valid BT and can be inserted at any node position without modifying surrounding logic. This allows modular mission building — a new "electronic warfare evasion" subtree can be inserted at the Threat Response Selector without touching the ISR mission logic. FSMs require adding transitions from every existing state to the new states.

3. Tick rate: BT ticks once per control cycle. For mission-level decisions: typically 10–50 Hz. For safety-critical conditions (battery, collision): 100–250 Hz. The tick rate determines decision latency — critical for the OODA loop cycle time.

8.5 Path Planning — From Current Position to Objective

Plain English First

Path planning answers the question: given a map of the environment and a goal location, what sequence of movements will get me there while avoiding obstacles, respecting vehicle constraints, and minimising some cost (time, fuel, exposure to threats)? For military UAVs this is not just obstacle avoidance — it includes terrain masking (flying low to stay below radar coverage), threat envelope avoidance (staying outside SAM engagement zones), and time-on-target constraints (arriving in a coordination window). Planners operate at two levels: global (full mission trajectory) and local (real-time reactive obstacle avoidance).

A* — Optimal Graph Search

A* is the foundation of global path planning. It searches a discretised map (occupancy grid or waypoint graph) for the minimum-cost path: Bestaoui Sebbane §10.1

A* Evaluation Function $$f(n) = g(n) + h(n)$$

$g(n)$: exact cost from start to node $n$ (known). $h(n)$: heuristic estimate of cost from $n$ to goal. Admissible heuristic: $h(n) \leq h^*(n)$ (never overestimates true cost). Common choices: Euclidean distance $h = \|\mathbf{p}_n - \mathbf{p}_{goal}\|_2$ (admissible if all moves have cost $\geq$ distance); Octile distance for 8-connected grids.

Complexity: $O(b^d)$ worst case, where $b$ is branching factor and $d$ is depth. On a 100×100×20 3D occupancy grid (typical indoor UAV planning): $\sim 200{,}000$ nodes. With admissible heuristic, A* is guaranteed to find the optimal path. For military terrain planning (10 km² area at 10 m resolution): $10^5$ nodes — typically solves in $<100$ ms with a good heuristic.

RRT* — Optimal Sampling-Based Planning

For continuous state spaces with complex kinodynamic constraints (UAV cannot turn instantaneously; minimum turning radius applies), grid-based A* is intractable. RRT* (Rapidly-Exploring Random Trees, optimal variant) samples random configurations and incrementally builds a tree: Bestaoui Sebbane §10.2

RRT* Core Algorithm (Asymptotically Optimal) $$x_{rand} \sim \text{Uniform}(\mathcal{X}) \quad \text{(sample random state)}$$ $$x_{nearest} = \arg\min_{x \in \mathcal{T}} \|x - x_{rand}\| \quad \text{(find nearest tree node)}$$ $$x_{new} = \text{Steer}(x_{nearest}, x_{rand}, \Delta) \quad \text{(extend toward sample)}$$ $$\text{Rewire}(\mathcal{T}, x_{new}, r_n) \quad \text{(optimise nearby connections)}$$

The Rewire step is what makes RRT* optimal (vs. RRT which is only complete): for each node $x$ within radius $r_n$ of $x_{new}$, if routing through $x_{new}$ gives a lower cost path from root to $x$, update the parent of $x$ to be $x_{new}$. As the number of samples $n \to \infty$, the path cost converges to the optimum. Rewire radius: $r_n = \gamma_{RRT^*}\left(\frac{\ln n}{n}\right)^{1/d}$ where $d$ is state space dimension — shrinks as more samples are added. Bestaoui Sebbane §10.2

Threat-Aware Planning — Cost Map Design

Military path planning adds threat cost fields to the standard obstacle map. Each SAM battery, radar, and known adversary position creates a cost gradient: MDPI Autonomous Control Ch.7

Composite Mission Cost Map $$c(\mathbf{p}) = c_{obs}(\mathbf{p}) + \sum_{k} w_k \cdot \phi_k(\mathbf{p})$$ $$\phi_{SAM}(\mathbf{p}) = \begin{cases} \infty & \|\mathbf{p} - \mathbf{p}_{SAM}\| \leq r_{kill} \\ \exp\!\left(-\frac{\|\mathbf{p}-\mathbf{p}_{SAM}\| - r_{kill}}{r_{threat}}\right) & r_{kill} < \|\mathbf{p}-\mathbf{p}_{SAM}\| \leq r_{detect} \\ 0 & \text{otherwise} \end{cases}$$

$r_{kill}$: SAM kill radius (hard exclusion zone — infinite cost); $r_{detect}$: SAM detection radius (soft avoidance zone — exponential cost); $r_{threat}$: cost decay length scale. The exponential tail allows the planner to "clip" through the outer detection zone if no other route exists — the cost is high but finite. Hard exclusion zones are never penetrated. Weight $w_k$ balances threat avoidance vs. mission efficiency.

Worked Example 8.2 — A* Path Cost with SAM Threat Field

A UAV must reach a target 80 km away. Two possible routes exist. Route A is direct (80 km), passing through a SAM outer detection zone for 15 km at mean cost $\phi = 2.5$ (exponential decay at 0.4× kill radius). Route B goes around (105 km), fully outside all threat zones ($\phi = 0$). Fuel cost per km = 1 unit. Threat cost weight $w = 8$. Which route has lower total cost?

Route A total cost: $c_A = 65 \times 1 + 15 \times (1 + 8 \times 2.5) = 65 + 15 \times 21 = 65 + 315 = \mathbf{380}$

Route B total cost: $c_B = 105 \times 1 + 0 = \mathbf{105}$

Decision: Route B (the long way round) has 3.6× lower cost. The planner correctly selects the safer route despite being 31% longer. If $w$ is reduced to 1 (mission commander accepts more risk): $c_A = 65 + 15 \times 3.5 = 117.5$ vs $c_B = 105$ — Route B still preferred, but only marginally. At $w \approx 0.7$, the routes become cost-equal — this is the risk tolerance threshold. This demonstrates how the weight $w$ encodes the mission risk appetite, and why it should be a configurable commander's parameter rather than a hard-coded value.

8.6 Autonomous Threat Response — Reactive Decision-Making

Plain English First

Global path planning works on the timescale of minutes. But threats can appear in seconds — a radar lock, a missile launch, a suddenly detected obstacle. Reactive threat response is the fast decision loop that operates independently of (and can override) the global mission plan. It runs at the same timescale as the OODA loop — 100 ms or faster. The key engineering challenge is ensuring these two decision layers (slow strategic + fast reactive) do not conflict — the reactive layer must be able to override the strategic layer without causing the UAV to take dangerous actions to "get back" to the plan.

Potential Field Reactive Avoidance

Artificial Potential Field (APF) methods compute a continuous force field over the workspace: goal attracts, obstacles repel. The control action is the gradient of this field: Bestaoui Sebbane §10.3

Artificial Potential Field — Attractive + Repulsive Forces $$U(\mathbf{p}) = U_{att}(\mathbf{p}) + \sum_i U_{rep,i}(\mathbf{p})$$ $$U_{att} = \frac{1}{2}k_{att}\|\mathbf{p} - \mathbf{p}_{goal}\|^2$$ $$U_{rep,i} = \begin{cases} \frac{1}{2}k_{rep}\left(\frac{1}{\rho_i} - \frac{1}{\rho_0}\right)^2 & \rho_i \leq \rho_0 \\ 0 & \rho_i > \rho_0 \end{cases}$$ $$\mathbf{F}(\mathbf{p}) = -\nabla U(\mathbf{p}) = \mathbf{F}_{att} + \sum_i \mathbf{F}_{rep,i}$$

$\rho_i = \|\mathbf{p} - \mathbf{p}_{obs,i}\|$: distance to obstacle $i$; $\rho_0$: obstacle influence radius; $k_{att}, k_{rep}$: attractive/repulsive gain constants. Known limitation: Local minima — the UAV can get trapped in a potential well between two obstacles. Solutions: random perturbation, hybrid APF + A* global planner. APF is used for fast reactive avoidance; A*/RRT* handles global planning.

Missile Evasion — Time-Critical Response

When a radar lock or missile launch is detected, the UAV must immediately execute an evasion manoeuvre that maximises miss distance while maintaining structural integrity. From Proportional Navigation (Module 03): the pursuer closes by maintaining constant LOS angle. The optimal evasion strategy is to maximise $\dot{\lambda}$ — the LOS rate: Bestaoui Sebbane §11.1

Optimal Evasion — Maximum LOS Rate $$\text{Max miss distance} \equiv \text{max } \dot{\lambda} \equiv \text{lateral acceleration perpendicular to LOS}$$ $$a_{evade} = a_{max} \cdot \hat{\mathbf{n}}_\perp \qquad \text{where } \hat{\mathbf{n}}_\perp \perp \mathbf{r}_{LOS}$$

The optimal evasion turn is always perpendicular to the line of sight between UAV and missile. This maximises $\dot{\lambda}$, forcing the PN guidance to command maximum lateral acceleration — potentially exceeding the missile's structural limits. For a small fast UAV (load factor limit 10g), the maximum structural acceleration provides ~4g lateral evasion. For a large MALE UAV (2–3g structural limit), evasion effectiveness is limited — which is why MALE platforms rely primarily on low RCS, chaff/flare dispensing, and route planning rather than kinetic evasion.

8.7 Ethics of Lethal Autonomous Weapons Systems (LAWS)

Plain English First

This section addresses a question that cannot be answered by equations alone: Should autonomous systems make life-and-death decisions without human authorisation? This is not merely a philosophical question for UAV engineers — it is a practical design constraint. The answer determines what your system is legally allowed to do, what certifications are required, and what happens when something goes wrong. Every engineer who designs autonomous UAV decision logic bears professional and potentially legal responsibility for those design choices, whether or not they ever think about it.

The IHL Requirements — Three Principles That Apply to Machines

International Humanitarian Law (IHL) applies to any weapons system, autonomous or not. Three principles directly constrain autonomous engagement decisions: MDPI Autonomous Control Ch.6

IHL PrincipleWhat It RequiresAutonomous System ChallengeTechnical Approach
Distinction Distinguish combatants from civilians and civilian objects Object classifiers are probabilistic, not certain. Classifier confidence ≠ legal certainty POMDP belief threshold + human confirmation for ambiguous cases
Proportionality Anticipated civilian harm must not be excessive relative to military advantage Requires estimating expected civilian casualties — machine cannot assign legal weight to this Current consensus: human must make proportionality judgments; machine provides data
Precaution Take all feasible precautions to minimise civilian harm before attacking POMDP information-gathering before engagement (Worked Example 8.1) partially addresses this Multi-observation thresholds; abort capability; return-and-report before engagement
The Accountability Gap When a human soldier makes a targeting error, there is a clear accountability chain. When a fully autonomous system makes a targeting error, who is accountable? The programmer? The commander who deployed it? The state? Current international law has no clear answer — this is the "accountability gap" at the centre of the LAWS debate. Engineers who design targeting logic are not legally protected by the fact that they were following orders or specifications. IHL Article 36 weapon review obligations apply to states, but the engineers who build the system carry professional and ethical responsibility that Article 36 does not extinguish. MDPI Autonomous Control Ch.6

The "Meaningful Human Control" Standard

The emerging international standard — advocated by the ICRC and 125+ states — is that humans must retain "meaningful control" over lethal decisions. Bestaoui Sebbane defines meaningful control as requiring that: Bestaoui Sebbane §11.2

  1. Contextual understanding: The human understands the operational context in which the system will engage (not just "authorised in zone X")
  2. Predictable performance: The system behaves predictably enough that the human's authorisation is genuinely meaningful (not a rubber stamp on opaque AI decisions)
  3. Temporal proximity: The authorisation is close enough in time to be relevant to the specific engagement (not a blanket authorisation days earlier)
  4. Abort capability: The human retains a meaningful ability to stop the engagement before it occurs

This standard creates concrete technical requirements: low-latency communication to the operator, explainable AI outputs (not just "confidence = 0.87" but "this looks like a tank because of track geometry and thermal signature"), and architectural abort pathways that cannot be bypassed by autonomous logic.

# PART 1 — POMDP Belief Updater

class TargetPOMDP:
    """
    POMDP for target identification before engagement.
    States: combatant (0) or civilian (1)
    Observations: threat_signal (0) or safe_signal (1)
    Bestaoui Sebbane §9.3
    """
    def __init__(self,
                 prior_combatant: float = 0.4,
                 obs_model: np.ndarray = None,
                 engage_threshold: float = 0.9):
        """
        prior_combatant : P(state=combatant) before any observation
        obs_model       : Z[s, o] = P(obs=o | state=s)
                          Row 0 = combatant, Row 1 = civilian
                          Col 0 = threat_signal, Col 1 = safe_signal
        engage_threshold: min belief in combatant to authorise engagement
        """
        self.belief = np.array([prior_combatant, 1 - prior_combatant])
        self.Z = obs_model if obs_model is not None else np.array([
            [0.85, 0.15],   # combatant: 85% threat signal, 15% safe signal
            [0.12, 0.88],   # civilian:  12% threat signal, 88% safe signal
        ])
        self.threshold = engage_threshold
        self.n_obs     = 0

    def update(self, observation: int) -> np.ndarray:
        """
        Bayesian belief update on new observation.
        observation: 0=threat_signal, 1=safe_signal
        Returns: updated belief [P(combatant), P(civilian)]
        Bestaoui Sebbane §9.3 Eq. 9.8
        """
        # Unnormalised: P(obs|s) * P(s)
        unnorm = self.Z[:, observation] * self.belief
        # Normalise
        self.belief = unnorm / (unnorm.sum() + 1e-12)
        self.n_obs += 1
        return self.belief.copy()

    def decision(self) -> str:
        """Current engagement recommendation based on belief threshold."""
        p_combatant = self.belief[0]
        if p_combatant >= self.threshold:
            return "ENGAGE"
        elif p_combatant <= 1 - self.threshold:
            return "NO THREAT — CLEAR"
        else:
            return f"GATHER MORE INFO  (need ≥{self.threshold:.0%} confidence)"


def demo_pomdp():
    print("\n" + "="*58)
    print("  POMDP TARGET IDENTIFICATION — Bestaoui Sebbane §9.4")
    print("="*58)
    pomdp = TargetPOMDP(prior_combatant=0.4, engage_threshold=0.9)
    print(f"\n  Prior: P(combatant)={pomdp.belief[0]:.3f}  Decision: {pomdp.decision()}")

    observations = [0, 0, 1, 0]   # threat, threat, safe, threat
    obs_names    = ['THREAT', 'SAFE']

    for i, obs in enumerate(observations):
        belief = pomdp.update(obs)
        print(f"  Obs {i+1}: {obs_names[obs]:<6} → P(combatant)={belief[0]:.4f}  "
              f"| {pomdp.decision()}")

    def __init__(self, grid_size: int = 40):
        self.G = grid_size
        self.grid = np.zeros((grid_size, grid_size))  # 0=free, 1=hard obstacle

    def add_sam_site(self, cx: int, cy: int,
                     r_kill: float, r_detect: float,
                     w_threat: float = 8.0):
        """
        Add SAM site threat field to the grid.
        r_kill   : hard exclusion radius (grid cells)
        r_detect : soft detection radius (exponential cost tail)
        w_threat : weight on threat cost vs distance cost
        """
        for y in range(self.G):
            for x in range(self.G):
                dist = np.sqrt((x-cx)**2 + (y-cy)**2)
                if dist <= r_kill:
                    self.grid[y, x] = 1e9   # hard exclusion — infinite cost
                elif dist <= r_detect:
                    threat_val = w_threat * np.exp(-(dist - r_kill) / (0.5*r_kill))
                    self.grid[y, x] += threat_val

    def plan(self, start: tuple, goal: tuple) -> list:
        """
        A* search from start to goal.
        Returns list of (x,y) waypoints from start to goal.
        """
        def heuristic(a, b):
            return np.sqrt((a[0]-b[0])**2 + (a[1]-b[1])**2)   # Euclidean

        open_set = [(heuristic(start, goal), 0, start)]
        came_from = {}
        g_cost = {start: 0}
        closed = set()
        neighbours = [(dx, dy) for dx in [-1,0,1] for dy in [-1,0,1] if (dx,dy) != (0,0)]

        while open_set:
            _, g, current = heapq.heappop(open_set)
            if current == goal:
                # Reconstruct path
                path = [current]
                while current in came_from:
                    current = came_from[current]
                    path.append(current)
                return list(reversed(path))

            if current in closed: continue
            closed.add(current)

            for dx, dy in neighbours:
                nx, ny = current[0]+dx, current[1]+dy
                if not (0 <= nx < self.G and 0 <= ny < self.G): continue
                if self.grid[ny, nx] >= 1e8: continue  # hard obstacle
                move_cost = np.sqrt(dx**2 + dy**2) + self.grid[ny, nx]
                new_g = g + move_cost
                neighbour = (nx, ny)
                if neighbour not in g_cost or new_g < g_cost[neighbour]:
                    g_cost[neighbour] = new_g
                    f = new_g + heuristic(neighbour, goal)
                    heapq.heappush(open_set, (f, new_g, neighbour))
                    came_from[neighbour] = current

        return []   # no path found

    def path_cost(self, path: list) -> float:
        if not path: return float('inf')
        cost = 0
        for i in range(1, len(path)):
            dx = path[i][0] - path[i-1][0]
            dy = path[i][1] - path[i-1][1]
            cost += np.sqrt(dx**2+dy**2) + self.grid[path[i][1], path[i][0]]
        return cost

class Status(Enum):
    SUCCESS = "SUCCESS"
    FAILURE = "FAILURE"
    RUNNING = "RUNNING"


class BTNode:
    """Base class for all BT nodes."""
    def tick(self, state: dict) -> Status:
        raise NotImplementedError


class Sequence(BTNode):
    """Execute children in order; fail on first child failure."""
    def __init__(self, children, name=""):
        self.children = children; self.name = name

    def tick(self, state):
        for child in self.children:
            result = child.tick(state)
            if result != Status.SUCCESS:
                return result
        return Status.SUCCESS


class Selector(BTNode):
    """Try children in order; succeed on first child success."""
    def __init__(self, children, name=""):
        self.children = children; self.name = name

    def tick(self, state):
        for child in self.children:
            result = child.tick(state)
            if result != Status.FAILURE:
                return result
        return Status.FAILURE


class Condition(BTNode):
    """Check a predicate on the state dict."""
    def __init__(self, predicate, name=""):
        self.pred = predicate; self.name = name

    def tick(self, state):
        result = Status.SUCCESS if self.pred(state) else Status.FAILURE
        return result


class Action(BTNode):
    """Execute a function and return its status."""
    def __init__(self, func, name=""):
        self.func = func; self.name = name

    def tick(self, state):
        return self.func(state)


def build_isr_bt():
    """Build the ISR mission behaviour tree from section 8.4."""
    # Actions
    def emergency_rtb(s):
        print(f"    [ACTION] EMERGENCY RTB — battery={s['battery']:.0%}")
        return Status.SUCCESS

    def transit_to_isr(s):
        print(f"    [ACTION] Transiting to ISR orbit")
        s['at_isr_orbit'] = True
        return Status.SUCCESS

    def capture_image(s):
        s['images_collected'] = s.get('images_collected', 0) + 1
        print(f"    [ACTION] Image captured (total: {s['images_collected']})")
        return Status.SUCCESS

    def transmit_data(s):
        if not s.get('comms_ok', True):
            print(f"    [ACTION] TRANSMIT FAILED — comms down")
            return Status.FAILURE
        print(f"    [ACTION] Data transmitted to GCS")
        return Status.SUCCESS

    def autonomous_loiter(s):
        print(f"    [ACTION] Switching to autonomous loiter")
        return Status.SUCCESS

    def evasive_manoeuvre(s):
        print(f"    [ACTION] Executing evasive manoeuvre — threat: {s.get('threat_type','unknown')}")
        s['threat_detected'] = False
        return Status.SUCCESS

    def continue_nominal(s):
        print(f"    [ACTION] Continuing nominal mission")
        return Status.SUCCESS

    # Build tree
    root = Selector([
        Sequence([                          # Emergency handler
            Condition(lambda s: s.get('battery',1.0) < 0.10, 'LowBattery'),
            Action(emergency_rtb, 'EmergencyRTB'),
        ], 'EmergencySeq'),

        Selector([                          # Mission fallback
            Sequence([                      # Primary mission
                Condition(lambda s: s.get('nav_valid', True), 'NavValid'),
                Condition(lambda s: s.get('fuel_ok', True), 'FuelOk'),
                Action(transit_to_isr, 'Transit'),
                Action(capture_image, 'CaptureImage'),
                Action(transmit_data, 'TransmitData'),
            ], 'PrimaryMission'),
            Sequence([                      # Degraded mode
                Condition(lambda s: not s.get('comms_ok', True), 'CommsLost'),
                Action(autonomous_loiter, 'AutoLoiter'),
            ], 'DegradedMode'),
        ], 'MissionSelector'),

        Selector([                          # Threat response
            Sequence([
                Condition(lambda s: s.get('threat_detected', False), 'ThreatDetected'),
                Action(evasive_manoeuvre, 'Evade'),
            ], 'EvasionSeq'),
            Action(continue_nominal, 'ContinueNominal'),
        ], 'ThreatResponse'),
    ], 'Root')

    return root
